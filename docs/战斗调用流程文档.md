# 战斗系统调用流程文档

> 基于实际代码梳理，所有文件路径相对于项目根目录 `GameNetty/`。

---

## 1. 战斗启动流程

### 1.1 C2M_StartBattleHandler — 开始战斗请求

**文件**: `Server/Hotfix/Demo/Battle/Handler/C2M_StartBattleHandler.cs`

**处理流程**:

1. 获取玩家所在 Map Scene
2. 获取或创建 `BattleRoomManagerComponent`（挂载在 Map Scene 上）
3. **重复战斗检查**: 调用 `roomManager.IsUnitInBattle(unit.Id)`，如果已在战斗中则返回错误码 `ERR_AlreadyInBattle`
4. **创建 BattleRoom**: 调用 `mapScene.AddChild<BattleRoom, int>(configId)`，configId 取自 `request.stageId` 或 `request.battleType`
5. **注册玩家**: 调用 `battleRoom.AddPlayer(unit.Id)`
6. **注册映射**: 通过 `roomManager` 建立 UnitId → BattleRoomId 的双向映射
7. **初始化战斗**: 调用 `battleRoom.InitBattle(unit, stageId, battleType)`
   - 内部通过 `UnitFactory.CreateHero()` 创建玩家英雄 BattleUnit（挂载 NumericComponent、NumericNoticeComponent、BattleMoveComponent、BattleUnitCombatComponent、BattleActionDecisionComponent、PlayerCombatModeComponent）
   - 读取 `WaveConfigCategory` 构建关卡配置（StageConfigInfo）
   - 创建 `WaveManagerComponent` 挂载到 BattleRoom
8. **设置状态**: `battleRoom.State = BattleState.Prepare`
9. **启动第一波**: 调用 `battleRoom.StartFirstWave().Coroutine()`
   - 先发送英雄创建消息 `M2C_CreateBattleUnits` 给所有玩家
   - 再调用 `WaveManagerComponent.StartFirstWave()` 生成第一波怪物

### 1.2 C2M_BattleReadyHandler — 战斗准备就绪

**文件**: `Server/Hotfix/Demo/Battle/Handler/C2M_BattleReadyHandler.cs`

**处理流程**:

1. 获取 Map Scene 上的 `BattleRoomManagerComponent`
2. 根据 unitId 查找对应的 `BattleRoom`
3. 验证 battleId 是否匹配
4. **切换状态**: `battleRoom.State = BattleState.Fighting`
5. **触发首次决策**: 遍历 `battleRoom.Units`，对每个存活单位的 `BattleActionDecisionComponent` 调用 `MakeDecision()`

### 1.3 BattleRoom 状态变化

**文件**: `Server/Model/Demo/Battle/BattleScene.cs`

`BattleState` 枚举定义:
- `None = 0` — 未初始化
- `Prepare = 1` — 准备阶段（StartBattle 后 → Ready 前）
- `Fighting = 2` — 战斗进行中（Ready 后）
- `WaitingChoice = 3` — 等待玩家选择（掉落/技能选择，当前代码中未使用）
- `Settle = 4` — 结算中（当前代码中未使用）
- `End = 5` — 已结束（胜利/失败后）

**BattleRoom.Awake** (`Server/Hotfix/Demo/Battle/BattleRoomSystem.cs`):
- 初始化 `State = BattleState.Prepare`、`SceneType = SceneType.Battle`
- 添加 `SlotManagerComponent`

**BattleRoom.Destroy**:
- 清空 PlayerIds 和 Units 字典
- 设置 `State = BattleState.End`

**BattleRoom 实现 IUpdate 接口**，因此每帧会被 ET 框架自动调用 `BattleRoomSystem.Update()`。

## 2. 每帧更新流程

### 2.1 BattleRoom.Update

**文件**: `Server/Hotfix/Demo/Battle/BattleRoomSystem.cs` — `BattleRoomSystem.Update()`

每帧执行逻辑:
1. **状态守卫**: 如果 `State != BattleState.Fighting`，直接 return
2. **波次状态守卫**: 获取 `WaveManagerComponent`，如果不存在或 `State != WaveState.Fighting`，直接 return
3. **遍历所有单位**: 遍历 `battleRoom.Units`，跳过 null 和已死亡的单位
4. **移动更新**: 对每个存活单位调用 `BattleMoveComponent.Update()`

### 2.2 BattleMoveComponent.Update

**文件**: `Server/Hotfix/Demo/Battle/BattleMoveComponentSystem.cs` — `BattleMoveComponentSystem.Update()`

处理逻辑:
1. **移动状态检查**: 如果 `IsMoving == false`，直接 return
2. **存活检查**: 如果 owner 为 null 或已死亡，调用 `StopMove()` 并 return
3. **到达检查**: 计算当前位置到 `TargetPosition` 的距离，如果 `< 0.01f`，调用 `StopMove()` 并 return
4. **计算移动量**: `moveDistance = moveSpeed × deltaTime / 1000`（deltaTime 为毫秒级帧间隔）
5. **更新位置**: 调用 `BattleDistanceHelper.MoveTowardsX()` 沿 X 轴移动
6. **微小移动过滤**: 如果实际位移 ≤ 0.0001f，不执行后续逻辑（避免精度抖动）
7. **到达判定**: 如果移动后距离目标 `< 0.01f`，调用 `StopMove()` 并发布 `ReachTargetEvent`

**关键**: 移动是一维的（仅 X 轴），这是卷轴战斗的布局方式。

### 2.3 决策在哪里被触发

决策 **不是** 在每帧 Update 中触发的，而是 **纯事件驱动** 的。决策触发点:

| 触发时机 | 触发位置 | 文件 |
|---------|---------|------|
| 战斗准备就绪 | `C2M_BattleReadyHandler` 遍历所有单位 | `Handler/C2M_BattleReadyHandler.cs` |
| 到达目标位置 | `BattleMoveComponentSystem.Update()` 发布 `ReachTargetEvent` | `BattleMoveComponentSystem.cs` → `Event/DecisionComponent_OnReachTarget.cs` |
| 冻结结束 | `FreezeComponentSystem.EndFreeze()` 发布 `FreezeEndEvent` | `FreezeComponentSystem.cs` → `Event/MoveComponent_OnFreezeEnd.cs` |
| 目标死亡 | `BattleUnitDead_Event` 遍历所有单位 | `Event/BattleUnitDead_Event.cs` |

## 3. 事件驱动的决策流程

### 3.1 DecisionComponent.MakeDecision 完整逻辑

**文件**: `Server/Hotfix/Demo/Battle/BattleActionDecisionComponentSystem.cs` — `BattleActionDecisionComponentSystem.MakeDecision()`

完整流程:

```
MakeDecision(owner)
  │
  ├─ 1. 前置检查: owner 为 null 或已死亡 → return
  │
  ├─ 2. 目标有效性检查: CurrentTarget 已死亡 → 置 null
  │
  ├─ 3. 调用 BattleSkillHelper.TrySelectBestAutoSkillPlan()
  │     │
  │     ├─ 返回 false（找不到可用技能计划）:
  │     │   ├─ 如果有旧目标 → 发布 RequestStopMoveEvent，清空 CurrentTarget
  │     │   └─ return
  │     │
  │     └─ 返回 true:
  │         ├─ 获取 newTarget = autoCastPlan.Target
  │         │
  │         ├─ 4. 目标变化检查:
  │         │   如果 currentTarget == null 或 ID 不同:
  │         │     ├─ 更新 self.CurrentTarget = newTarget
  │         │     └─ 发布 TargetChangedEvent { UnitId, OldTargetId, NewTargetId }
  │         │
  │         ├─ 5. 范围判断: BattleSkillHelper.IsInSkillRange()
  │         │   │
  │         │   ├─ 在范围内:
  │         │   │   ├─ 发布 RequestStopMoveEvent
  │         │   │   └─ 发布 RequestCastEvent { Unit, SkillId, TargetId }
  │         │   │
  │         │   └─ 不在范围内:
  │         │       ├─ 计算 BattleSkillHelper.ComputeDesiredCastPosition()
  │         │       └─ 发布 RequestMoveEvent { Unit, TargetPosition }
  │         │
  │         └─ 完成
```

### 3.2 事件链详细说明

#### TargetChangedEvent（目标变化事件）

**定义**: `Server/Model/Demo/Battle/BattleEvents.cs`

**发布位置**: `BattleActionDecisionComponentSystem.MakeDecision()` — 当决策组件切换锁定目标时

**处理器**: ⚠️ **无处理器** — 代码中只发布了该事件，但没有找到 `AEvent<BattleRoom, TargetChangedEvent>` 类的实现。该事件目前仅作为通知，没有订阅者。

#### RequestMoveEvent（请求移动事件）

**定义**: `Server/Model/Demo/Battle/BattleEvents.cs`

**发布位置**: `BattleActionDecisionComponentSystem.MakeDecision()` — 当目标不在施法范围内时

**处理器**: `Server/Hotfix/Demo/Battle/Event/MoveComponent_OnRequestMove.cs` — `MoveComponent_OnRequestMove`
- 获取单位的 `BattleMoveComponent`
- 调用 `moveComponent.StartMove(args.TargetPosition)`
- `StartMove()` 内部: 设置 IsMoving=true、TargetPosition、MoveSpeed，然后通过 `BattleUnitHelper.BroadcastMoveCommand()` 广播移动指令给客户端

#### RequestStopMoveEvent（请求停止移动事件）

**定义**: `Server/Model/Demo/Battle/BattleEvents.cs`

**发布位置**: `BattleActionDecisionComponentSystem.MakeDecision()` — 目标无效或在施法范围内时

**处理器**: `Server/Hotfix/Demo/Battle/Event/MoveComponent_OnRequestStopMove.cs` — `MoveComponent_OnRequestStopMove`
- 获取单位的 `BattleMoveComponent`
- 调用 `moveComponent.StopMove()`
- `StopMove()` 内部: 设置 IsMoving=false，通过 `BattleUnitHelper.BroadcastMoveCommand()` 广播停止移动

#### RequestCastEvent（请求施法事件）

**定义**: `Server/Model/Demo/Battle/BattleEvents.cs`

**发布位置**: `BattleActionDecisionComponentSystem.MakeDecision()` — 当目标在施法范围内时

**处理器**: ⚠️ **无处理器** — 代码中搜索 `AEvent.*RequestCastEvent` 没有找到任何匹配。这意味着 **自动战斗的施法指令发布后没有任何组件消费它，自动战斗实际上无法完成施法**。这是一个 Bug。

> **补充说明**: 客户端手动施法走的是 `C2M_CastSkillHandler`（`Server/Hotfix/Demo/Battle/Handler/C2M_CastSkillHandler.cs`）和 `C2M_AttackTargetHandler`（`Server/Hotfix/Demo/Battle/Handler/C2M_AttackTargetHandler.cs`），它们直接调用 `BattleSkillHelper.TryExecuteSkill()` 和 `BattleSkillHelper.TryExecuteNormalAttack()`，不经过事件系统。但自动战斗模式下，`RequestCastEvent` 没有任何订阅者来执行实际的技能逻辑。

#### ReachTargetEvent（到达目标位置事件）

**定义**: `Server/Model/Demo/Battle/BattleEvents.cs`

**发布位置**: `BattleMoveComponentSystem.Update()` — 移动组件到达目标位置时

**处理器**: `Server/Hotfix/Demo/Battle/Event/DecisionComponent_OnReachTarget.cs` — `DecisionComponent_OnReachTarget`
- 获取单位的 `BattleActionDecisionComponent`
- 调用 `decisionComponent.MakeDecision()` → 重新进入决策流程（此时应该已在施法范围内，会发布 `RequestCastEvent`）

## 4. 施法流程

### 4.1 BattleSkillHelper.TrySelectBestAutoSkillPlan

**文件**: `Server/Hotfix/Demo/Battle/BattleSkillHelper.cs`

完整流程:

```
TrySelectBestAutoSkillPlan(caster, preferredTarget, out plan)
  │
  ├─ 1. 前置检查: caster 为 null/已死亡/battleRoom 为 null/combat 为 null → return false
  │
  ├─ 2. 获取 UnitCombatConfig → AutoBattleStrategyConfig
  │
  ├─ 3. 获取自动技能列表 GetAutoSkillIds()
  │     从 unitCombatConfig.AutoSkillIds 收集 + unitCombatConfig.NormalAttackSkillId 追加
  │     如果列表为空 → return false
  │
  ├─ 4. 第一次尝试: TryBuildAutoCastPlan(readyOnly=true)
  │     只考虑 CD 已就绪的技能
  │     │
  │     ├─ 成功 → return true
  │     │
  │     └─ 失败:
  │         ├─ 如果 strategyConfig.AllowPreMoveOnCooldown == false → return false
  │         └─ 否则继续
  │
  └─ 5. 第二次尝试: TryBuildAutoCastPlan(readyOnly=false)
        允许 CD 中但可以预移动的技能
```

**TryBuildAutoCastPlan 内部逻辑**:

```
TryBuildAutoCastPlan(caster, battleRoom, combat, autoSkillIds, strategyConfig, preferredTarget, readyOnly, out bestPlan)
  │
  ├─ 遍历 autoSkillIds:
  │   ├─ 检查 CanUseSkillForAuto():
  │   │   ├─ skillConfig 为 null → skip
  │   │   ├─ !skillConfig.IsEnabled → skip
  │   │   ├─ castCheckConfig.CheckCanAttack && !combat.CanAttack → skip
  │   │   └─ readyOnly && castCheckConfig.CheckCooldown && !combat.IsSkillReady → skip
  │   │
  │   └─ 遍历 battleRoom.Units:
  │       ├─ 检查 IsTargetValid(): 阵营匹配 + 存活检查
  │       ├─ 计算距离、期望施法距离、期望施法位置
  │       ├─ 构建 AutoCastPlan 候选
  │       └─ 通过 IsBetterAutoCastPlan() 与当前最优比较
  │
  └─ 返回是否找到可行计划
```

**IsBetterAutoCastPlan 优先级比较**（从高到低）:
1. **当前目标偏好** (`PreferCurrentTarget`): 如果 preferredTarget 是候选目标且距离在容忍度内
2. **目标类型权重**: Boss(+200) > Elite(+100) > 普通(0)
3. **技能选择规则**: `HighestPriorityThenMove` 或 `ShortestMoveThenPriority`
4. **目标选择规则**: `NearestEnemy`（默认）或 `LowestHp`
5. **移动距离**: 距离短者优先
6. **施法距离**: 离目标近者优先
7. **技能优先级**: `skillConfig.Priority` 高者优先

### 4.2 BattleSkillHelper.TryExecuteSkill

**文件**: `Server/Hotfix/Demo/Battle/BattleSkillHelper.cs`

完整流程:

```
TryExecuteSkill(caster, skillId, explicitTargetId, out result, ignoreAutoModeLimit)
  │
  ├─ 1. 前置检查: caster 为 null/已死亡 → ERR_UnitIsDead
  ├─ 2. BattleRoom 检查 → ERR_BattleNotFound
  ├─ 3. CombatComponent 检查 → ERR_BattleAttackNotReady
  ├─ 4. SkillConfig 检查: 为 null 或 !IsEnabled → ERR_SkillNotFound
  ├─ 5. 关联配置完整性检查: TargetingConfig / CastCheckConfig / EffectGroupConfig
  ├─ 6. 设置 AttackCooldown 和 AttackRange
  ├─ 7. CanAttack 检查 → ERR_BattleAttackNotReady
  ├─ 8. 技能 CD 检查: IsSkillReady() → ERR_SkillCooldown
  ├─ 9. 自动模式限制检查 → ERR_BattleInAutoMode
  │
  ├─ 10. 选目标: SelectTargets()
  │     ├─ 如果有 explicitTargetId:
  │     │   ├─ 验证目标有效性（阵营/存活）
  │     │   ├─ 验证距离范围
  │     │   └─ 直接返回单目标列表
  │     │
  │     ├─ 如果需要显式目标 (NeedExplicitTarget || SelectType==LockedTarget):
  │     │   └─ 返回错误 "Explicit target required"
  │     │
  │     └─ 自动选目标:
  │         ├─ SelectType == AllEnemiesInRange: 收集范围内所有有效目标
  │         ├─ SelectType == NearestEnemy: 选最近的有效目标
  │         ├─ 排序: SortRule (None/Nearest/LowestHp)
  │         └─ 限制: MaxTargetCount
  │
  ├─ 11. 广播施法: BattleUnitHelper.BroadcastSkillCast()
  │
  ├─ 12. 对每个目标 ApplyEffects():
  │     遍历 SkillEffectGroupConfig.EffectIds:
  │     ├─ EffectType == Damage:
  │     │   ├─ CalculateDamage(): 公式 = BaseValue + Attack × RatioAtk - Defense × RatioDef
  │     │   ├─ 限制 MinValue/MaxValue
  │     │   ├─ target.TakeDamage(damage)
  │     │   ├─ BattleUnitHelper.BroadcastDamage()
  │     │   └─ 如果击杀: BattleUnitHelper.BroadcastUnitDead()
  │     └─ 其他效果类型: 当前 TryExecuteSkill 中只处理 Damage
  │
  ├─ 13. 设置技能 CD: combat.StartSkillCooldown()
  │
  └─ 14. 返回 SkillExecutionResult { SkillId, MainTarget, HitTargetsCount, TotalDamage, CooldownEnd }
```

> **注意**: `TryExecuteSkill` 中的效果处理只处理了 `EffectTypeDamage`。另外有一套独立的 `EffectApplyComponent.ApplyEffects()` 方法（`Server/Hotfix/Demo/Battle/EffectApplyComponentSystem.cs`）支持更多效果类型（Damage、Freeze、Knockback、Heal），通过发布事件来解耦。两套系统目前并行存在。

### 4.3 RequestCastEvent 处理器分析

⚠️ **关键问题**: `RequestCastEvent` **没有事件处理器**。

- 事件定义在 `Server/Model/Demo/Battle/BattleEvents.cs`
- 事件发布在 `Server/Hotfix/Demo/Battle/BattleActionDecisionComponentSystem.cs` 的 `MakeDecision()` 方法中
- 通过 `EventSystem.Instance.Publish<BattleRoom, RequestCastEvent>(...)` 发布
- **但全项目没有 `AEvent<BattleRoom, RequestCastEvent>` 的实现类**

这意味着: **在自动战斗模式下，单位移动到目标附近后，`MakeDecision()` 会发布 `RequestCastEvent`，但没有任何代码消费该事件来执行实际的技能逻辑。自动战斗的施法环节是断裂的。**

## 5. 波次管理流程

### 5.1 WaveManagerComponent 生命周期

**Model 文件**: `Server/Model/Demo/Battle/WaveManagerComponent.cs`

**System 文件**: `Server/Hotfix/Demo/Battle/WaveManagerComponentSystem.cs`

**创建时机**: `BattleRoomSystem.InitBattle()` → `battleRoom.AddComponent<WaveManagerComponent, int, List<int>>(stageId, waveConfigIds)`

**Awake 初始化**:
- `StageConfigId` = 传入的 stageId
- `WaveConfigIds` = 所有波次配置 ID 列表（按 WaveNumber 排序）
- `TotalWaves` = 波次总数
- `CurrentWaveIndex = -1`（尚未开始）
- `State = WaveState.None`
- `WaveInterval = 5000`（毫秒，波次间隔）
- `AutoStartNextWave = true`

**Destroy**: 清空 CurrentWaveMonsterIds 和 WaveConfigIds

**WaveState 枚举**:
- `None = 0` — 未初始化
- `Preparing = 1` — 波次准备中
- `Fighting = 2` — 波次战斗中
- `Completed = 3` — 波次完成

### 5.2 完整波次流程

```
StartFirstWave()
  └─ StartNextWave()
       │
       ├─ 边界检查: CurrentWaveIndex >= TotalWaves - 1 → OnAllWavesCompleted()（胜利）
       │
       ├─ CurrentWaveIndex++
       ├─ State = WaveState.Preparing
       │
       ├─ 广播 M2C_WaveStart { battleId, waveNumber, totalWaves, monsterCount }
       │
       ├─ SpawnWaveMonsters(waveConfigId)
       │   │
       │   └─ SpawnWaveMonstersFromBatches(waveConfig)
       │       │
       │       ├─ 遍历 waveConfig.Batches:
       │       │   ├─ 获取 SpawnConfig (batch.SpawnId)
       │       │   ├─ 如果 batch.Delay > 0 → 等待 delay 毫秒
       │       │   └─ SpawnFromSpawnConfig(spawnConfig)
       │       │       ├─ 遍历 spawnConfig.Monsters:
       │       │       │   ├─ 随机 X 偏移: center ± spreadRange
       │       │       │   ├─ UnitFactory.CreateMonster() 创建怪物
       │       │       │   ├─ 注册到 battleRoom.Units
       │       │       │   ├─ 添加到 CurrentWaveMonsterIds
       │       │       │   └─ 收集 BattleUnitInfo
       │       │       └─ 广播 M2C_CreateBattleUnits
       │       │
       │       └─ 完成
       │
       └─ State = WaveState.Fighting
         WaveStartTime = 当前时间
```

### 5.3 OnMonsterDead（怪物死亡回调）

**调用链**: `BattleUnit.TakeDamage()` → HP ≤ 0 → 发布 `BattleUnitDead` → `BattleUnitDead_Event.Run()` → 判断 `deadUnit.Camp == UnitCamp.Enemy` → `waveManager.OnMonsterDead(monsterId)`

**文件**: `Server/Hotfix/Demo/Battle/WaveManagerComponentSystem.cs` — `WaveManagerComponentSystem.OnMonsterDead()`

```
OnMonsterDead(monsterId)
  │
  ├─ 从 CurrentWaveMonsterIds 移除该怪物 ID
  │
  └─ 如果 CurrentWaveMonsterIds.Count == 0 && State == WaveState.Fighting:
      └─ OnWaveCompleted()
```

### 5.4 OnWaveCompleted（波次完成）

```
OnWaveCompleted()
  │
  ├─ State = WaveState.Completed
  ├─ 计算波次持续时间
  │
  ├─ 广播 M2C_WaveComplete { battleId, waveNumber, totalWaves, duration }
  │
  ├─ 如果还有下一波 (CurrentWaveIndex < TotalWaves - 1):
  │   ├─ AutoStartNextWave == true → StartNextWave()（自动开始）
  │   └─ AutoStartNextWave == false → （等待手动触发，当前无手动触发接口）
  │
  └─ 如果已是最后一波 → OnAllWavesCompleted()（胜利）
```

### 5.5 OnAllWavesCompleted（全部波次完成 → 胜利）

```
OnAllWavesCompleted()
  │
  ├─ battleRoom.State = BattleState.End
  │
  ├─ 广播 M2C_BattleEnd { battleId, success=true, duration=0 }
  │
  ├─ 等待 5000 毫秒
  │
  ├─ 清理映射:
  │   ├─ roomManager.RemoveUnitFromBattleRoom(playerId) 对所有玩家
  │   └─ roomManager.RemoveBattleRoom(battleRoom.Id)
  │
  └─ battleRoom.Dispose()
```

## 6. 特殊事件处理

### 6.1 死亡事件链

**文件**: `Server/Hotfix/Demo/Battle/BattleUnitSystem.cs` — `BattleUnitSystem.TakeDamage()`

```
TakeDamage(damage)
  │
  ├─ 已死亡 → return
  ├─ currentHp -= damage（不低于 0）
  ├─ numeric.Set(NumericType.Hp, newHp)
  │
  └─ if newHp <= 0:
      ├─ self.IsDead = true
      └─ EventSystem.Instance.Publish(new BattleUnitDead { BattleUnit = self })
```

**BattleUnitDead 事件处理器**: `Server/Hotfix/Demo/Battle/Event/BattleUnitDead_Event.cs` — `BattleUnitDead_Event.Run()`

```
BattleUnitDead_Event.Run(scene, args)
  │
  ├─ 获取 deadUnit → BattleRoom
  │
  ├─ 通知所有以该单位为目标的角色重新决策:
  │   遍历 battleRoom.Units:
  │     如果 unit.CurrentTarget.Id == deadUnit.Id:
  │       └─ decisionComponent.MakeDecision()
  │
  ├─ 释放站位插槽:
  │   SlotManagerComponent.ReleaseSlot(deadUnit.Id)
  │   SlotManagerComponent.ReleaseAllSlotsForTarget(deadUnit.Id)
  │
  ├─ 如果 deadUnit 是敌方 (UnitCamp.Enemy):
  │   └─ waveManager.OnMonsterDead(deadUnit.Id)
  │       └─ → OnWaveCompleted() → 可能触发 StartNextWave() 或 OnAllWavesCompleted()
  │
  └─ 如果 deadUnit 是友方 (UnitCamp.Friend):
      ├─ 检查是否所有英雄已死亡
      └─ 如果全部死亡 → OnBattleFailed()
          ├─ battleRoom.State = BattleState.End
          ├─ 广播 M2C_BattleEnd { success=false }
          ├─ 等待 3000 毫秒
          ├─ 清理映射
          └─ battleRoom.Dispose()
```

### 6.2 冻结事件链

冻结涉及三个事件和一个组件:

**事件定义**: `Server/Model/Demo/Battle/BattleEvents.cs`
- `FreezeEvent` — 冻结请求
- `FreezeStartEvent` — 冻结开始通知（用于移动组件中断）
- `FreezeEndEvent` — 冻结结束通知（用于恢复决策）

**组件**: `Server/Model/Demo/Battle/FreezeComponent.cs` + `Server/Hotfix/Demo/Battle/FreezeComponentSystem.cs`

#### 冻结触发（两种路径）

**路径 A**: 通过 `EffectApplyComponent`（`Server/Hotfix/Demo/Battle/EffectApplyComponentSystem.cs`）
```
EffectApplyComponent.ApplySingleEffect() [EffectType == Freeze]
  └─ PublishFreezeEvent(target, sourceId, durationMs)
      └─ EventSystem.Instance.Publish(FreezeEvent)
          └─ FreezeEvent_OnFreeze.Run()
              └─ target.AddComponent<FreezeComponent>() (如不存在)
              └─ freezeComp.ApplyFreeze(durationMs)
```

**路径 B**: 直接调用 `FreezeComponent.ApplyFreeze()`

#### ApplyFreeze 执行流程

**文件**: `Server/Hotfix/Demo/Battle/FreezeComponentSystem.cs` — `FreezeComponentSystem.ApplyFreeze()`

```
ApplyFreeze(durationMs)
  │
  ├─ 已冻结 → return
  │
  ├─ 发布 FreezeStartEvent → MoveComponent_OnFreezeStart:
  │   └─ moveComp.StopMove()（停止移动）
  │
  ├─ self.IsFrozen = true
  ├─ self.FreezeEndTime = 当前时间 + durationMs
  │
  ├─ 注册一次性定时器 (TimerInvokeType.FreezeEnd):
  │   └─ 到时间 → FreezeEndTimer.Run() → FreezeComponent.EndFreeze()
  │
  └─ 广播 M2C_UnitFrozen { unitId, durationMs } 给客户端
```

#### EndFreeze 执行流程

```
EndFreeze()
  │
  ├─ 未冻结 → return
  │
  ├─ self.IsFrozen = false
  ├─ self.FreezeEndTime = 0
  │
  └─ 发布 FreezeEndEvent → MoveComponent_OnFreezeEnd:
      └─ decisionComponent.MakeDecision()（重新决策）
```

#### 冻结完整事件链

```
FreezeEvent (冻结请求)
  └─ FreezeEvent_OnFreeze.Run()                    [Server/Hotfix/Demo/Battle/Event/FreezeEvent_OnFreeze.cs]
      └─ FreezeComponent.ApplyFreeze()
          │
          ├─ FreezeStartEvent (冻结开始)
          │   └─ MoveComponent_OnFreezeStart.Run()  [Server/Hotfix/Demo/Battle/Event/MoveComponent_OnFreezeStart.cs]
          │       └─ moveComp.StopMove()
          │
          ├─ 注册定时器
          │   └─ FreezeEndTimer.Run()               [Server/Hotfix/Demo/Battle/FreezeComponentSystem.cs]
          │       └─ FreezeComponent.EndFreeze()
          │           │
          │           └─ FreezeEndEvent (冻结结束)
          │               └─ MoveComponent_OnFreezeEnd.Run() [Server/Hotfix/Demo/Battle/Event/MoveComponent_OnFreezeEnd.cs]
          │                   └─ decisionComponent.MakeDecision()
          │
          └─ 广播 M2C_UnitFrozen
```

### 6.3 击退事件链

**事件定义**: `Server/Model/Demo/Battle/BattleEvents.cs` — `KnockbackEvent { Target, Attacker, Distance, Direction }`

**触发路径**: 通过 `EffectApplyComponent.ApplySingleEffect()` [EffectType == Knockback]

```
EffectApplyComponent.PublishKnockbackEvent(target, caster, distance, direction)
  └─ EventSystem.Instance.Publish(KnockbackEvent)
      └─ KnockbackEvent_OnKnockback.Run()          [Server/Hotfix/Demo/Battle/Event/KnockbackEvent_OnKnockback.cs]
```

**处理器**: `Server/Hotfix/Demo/Battle/Event/KnockbackEvent_OnKnockback.cs`

```
KnockbackEvent_OnKnockback.Run(scene, args)
  │
  ├─ target 为 null 或已死亡 → return
  ├─ moveComp 为 null → return
  │
  ├─ moveComp.StopMove()（停止当前移动）
  │
  ├─ 计算击退位置:
  │   newPosition = target.Position + Direction × Distance
  │   target.Position = newPosition（直接瞬移，无动画过程）
  │
  └─ BattleUnitHelper.BroadcastKnockback(target, distance, direction)
      └─ 广播 M2C_UnitKnockback { unitId, distance, direction, newPosition }
```

> **注意**: 击退后没有触发重新决策。被击退的单位如果正在自动战斗，它的决策状态不会立即更新，需要等到下一次决策触发点（如到达目标、目标死亡等）才会重新决策。

## 附录：关键问题汇总

### ⚠️ Bug 1: RequestCastEvent 无处理器

**严重程度**: 高

`BattleActionDecisionComponentSystem.MakeDecision()` 在目标进入施法范围后会发布 `RequestCastEvent`，但项目中没有 `AEvent<BattleRoom, RequestCastEvent>` 的实现。这意味着 **自动战斗的施法逻辑永远不会被执行**。

**建议**: 需要创建一个 `AEvent<BattleRoom, RequestCastEvent>` 处理器，在其中调用 `BattleSkillHelper.TryExecuteSkill()` 来执行实际施法。

### ⚠️ Bug 2: TargetChangedEvent 无处理器

**严重程度**: 低

`TargetChangedEvent` 被发布但没有订阅者。如果未来需要在目标切换时执行逻辑（如切换动画、打断当前动作等），需要添加处理器。

### ⚠️ 问题 3: 击退后未触发重新决策

**严重程度**: 低

击退处理器（`KnockbackEvent_OnKnockback`）修改了单位位置但没有触发 `MakeDecision()`。被击退后可能需要重新评估距离和目标。

### ⚠️ 问题 4: duration 字段始终为 0

**严重程度**: 低

胜利和失败时 `M2C_BattleEnd.duration` 都硬编码为 0，没有计算实际战斗时长。

### ⚠️ 问题 5: 两套效果系统并存

**严重程度**: 中

`BattleSkillHelper.TryExecuteSkill()` 内部直接处理 Damage 效果，而 `EffectApplyComponent.ApplyEffects()` 通过事件系统处理 Damage/Freeze/Knockback/Heal。两套系统逻辑重复且行为不一致。
