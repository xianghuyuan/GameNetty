本文档基于当前代码实现，整理战斗系统的运行结构、职责边界与扩展点，重点说明维护时最常接触的 BattleRoom、BattleUnit 以及各战斗组件的分工。

## 2. 核心实体关系

```text
BattleRoom
 ├─ PlayerIds
 ├─ WaveManagerComponent
 └─ Units[BattleUnit]
     ├─ NumericComponent               # 数值：HP、攻击、防御、速度
     ├─ BattleUnitCombatComponent      # 战斗冷却与技能CD管理
     ├─ BattleActionDecisionComponent  # 行为决策（选目标、选技能、追击/攻击状态）
     ├─ BattleMoveComponent            # 移动执行（固定点/跟随/停止/击退）
     ├─ BattleAIComponent              # 自动战斗总入口（友方/敌方均挂载）
     ├─ EffectApplyComponent           # 效果应用（伤害/冻结/击退/治疗）
     ├─ FreezeComponent                # 冻结状态管理（按需挂载）
     └─ PlayerCombatModeComponent      # 玩家单位专属：手动/自动模式控制
```

### 2.1 BattleRoom

- 战斗运行容器，持有房间内所有 `BattleUnit`。
- 提供玩家列表与战斗单位字典，是服务端判定和广播的挂载点。
- `BattleRoomSystem.Update` 中轮询所有存活单位的移动（`BattleMoveComponent.Update`）和 AI 行为（`BattleAIComponent.Update`）。

### 2.2 BattleUnit

- 表示战斗中的实际单位实例，挂载于 `BattleRoom`。
- 维护阵营（`Camp`）、位置（`Position`）、死亡状态（`IsDead`）、配置 ID 等基础数据。
- 实际数值依赖 `NumericComponent`，战斗行为依赖各战斗组件。
- 怪物和玩家英雄均为 `BattleUnit`，通过 `Camp` 区分阵营：
  - `UnitCamp.Friend`：玩家英雄
  - `UnitCamp.Enemy`：怪物

## 3. 三层行为拆分

当前战斗行为拆成"自动战斗入口 → 决策 → 移动执行"三层。

### 3.1 BattleAIComponent：自动战斗总入口

对应代码：`BattleAIComponentSystem.Update`

职责：

- 作为自动战斗的统一入口，英雄和怪物均挂载此组件。
- 判断单位是否有效、是否死亡。
- 对友方单位（英雄）额外检查 `PlayerCombatModeComponent.IsAutoBattle`；未开启自动战斗时停止移动并重置决策状态。
- 满足条件时，把控制权交给 `BattleActionDecisionComponent.Update()`。

结论：`BattleAIComponent` 不直接做选目标、算距离、放技能，只负责"该不该进入自动战斗"。

### 3.2 BattleActionDecisionComponent：行为判定层

对应代码：`BattleActionDecisionComponentSystem.Update`

职责：

- 基于当前单位与房间上下文生成自动施法方案。
- 维护 `CurrentTarget` 与当前状态（`Idle` / `Chasing` / `Attacking`）。
- 根据 `BattleSkillHelper.TrySelectBestAutoSkillPlan` 的结果做状态切换。

判定流程：

```text
Update
 ├─ 检查 owner / combat 是否有效
 ├─ 调 BattleSkillHelper.TrySelectBestAutoSkillPlan
 │   ├─ 选技能、选目标
 │   └─ 算出需要移动距离与理想施法距离
 ├─ 无方案 -> Reset + StopMove + Idle
 ├─ 已在技能范围内 -> StopMove + Attacking + TryAttack
 ├─ 切目标但距离落在死区 -> 保持 Attacking，不立刻追击抖动
 └─ 否则 -> Chasing + FollowTarget
```

关键设计点：

- **技能驱动决策**：先决定"最优技能计划"，再决定是否追击。
- **死区控制**：`NearbyTargetSwitchDeadZone` 避免目标切换时反复停走抖动。
- **状态跟踪**：`LastTrace*` 字段和心跳节流位可用于排查"为什么一直追击 / 为什么没放技能"（详细 Log 当前已注释，需要时可恢复）。

### 3.3 BattleMoveComponent：移动执行层

对应代码：`BattleMoveComponentSystem`

职责：

- 执行固定点移动和 FollowTarget 跟随移动。
- 维护目标点、移动速度、跟随目标 ID、期望距离、站位方向等运行时状态。
- 保存被中断的移动命令（`InterruptedCommand`），冻结结束后用于恢复。
- 在必要时广播 `M2C_BattleUnitMoveCommand` 给客户端。

核心方法：

| 方法 | 说明 |
|------|------|
| `StartMove(Vector3)` | 固定点移动，目标变化超阈值时才广播 |
| `FollowTarget(BattleUnit, float)` | 追击跟随，自动计算站位方向，状态变化时才广播 |
| `StopMove()` | 停止移动，有激活命令时广播 `isMoving=false` |
| `Knockback(float, float)` | 瞬间击退，广播 `M2C_UnitKnockback` |
| `Update()` | 逐帧推进位置，跟随模式下实时重算目标点 |

## 4. 战斗属性与冷却：BattleUnitCombatComponent

对应代码：`BattleUnitCombatComponentSystem`

职责：

- 管理攻击冷却（`AttackCooldown`）和上次攻击时间（`LastAttackTime`）。
- 维护技能冷却组字典 `SkillCooldownEnds`，支持冷却组共享 CD（`CooldownGroupId`）。
- 提供 `IsAttackReady()`、`IsSkillReady(SkillConfig)`、`StartSkillCooldown(SkillConfig)` 等查询接口。
- 所有时间均使用 `ServerFrameTime()`，与服务端帧时间一致。

## 5. 效果应用：EffectApplyComponent

对应代码：`EffectApplyComponentSystem`

职责：

- 接收 `SkillEffectGroupConfig` 并逐个应用效果，返回 `List<EffectResult>`。
- 支持效果类型（`EffectType`）：
  - `Damage`：计算伤害公式并发布 `DamageEvent`
  - `Freeze`：发布 `FreezeEvent`
  - `Knockback`：发布 `KnockbackEvent`
  - `Heal`：直接调用 `BattleUnit.Heal`
- 伤害计算支持"攻击减防御"公式（`FormulaType=1`），并应用最小/最大值限制。

## 6. 冻结状态：FreezeComponent

对应代码：`FreezeComponentSystem`

职责：

- 维护单位的冻结状态（`IsFrozen`、`FreezeEndTime`）。
- `ApplyFreeze(durationMs)` 流程：
  1. 发布 `FreezeStartEvent` → `MoveComponent_OnFreezeStart` 保存并中断当前移动。
  2. 设置冻结状态和结束时间。
  3. 注册 `FreezeEndTimer`（`TimerInvokeType.FreezeEnd`）定时结束。
  4. 广播 `M2C_UnitFrozen` 给客户端。
- `EndFreeze()` 清除冻结状态，发布 `FreezeEndEvent` → `MoveComponent_OnFreezeEnd` 恢复之前的移动命令。
- 此组件为按需挂载，首次冻结时由 `FreezeEvent_OnFreeze` 自动添加到目标单位。

## 7. 事件系统在战斗中的应用

战斗效果通过 `EventSystem` 发布/订阅，实现组件间解耦：

| 事件 | 发布方 | 订阅方 | 作用 |
|------|--------|--------|------|
| `DamageEvent` | `EffectApplyComponent` | `DamageEvent_OnDamage` | 扣血、广播伤害消息 |
| `FreezeEvent` | `EffectApplyComponent` | `FreezeEvent_OnFreeze` | 添加/应用冻结组件 |
| `FreezeStartEvent` | `FreezeComponent` | `MoveComponent_OnFreezeStart` | 中断当前移动，保存命令到 `InterruptedCommand` |
| `FreezeEndEvent` | `FreezeComponent` | `MoveComponent_OnFreezeEnd` | 恢复被中断的移动命令 |
| `KnockbackEvent` | `EffectApplyComponent` | `KnockbackEvent_OnKnockback` | 执行击退 |
| `BattleUnitDead` | `BattleUnit.TakeDamage` | `BattleUnitDead_Event` | 怪物死亡通知波次管理器；英雄全灭触发战斗失败 |

## 8. 技能系统在架构中的位置

`BattleSkillHelper` 是战斗决策的关键公共层，既服务自动战斗也服务主动施法。

主要职责：从配置表读取技能定义，进行施法校验、冷却校验、目标选择、距离判定、伤害结算与死亡广播。

`BattleActionDecisionComponent` 只消费 `AutoCastPlan` 结果，不关心技能细节；`BattleMoveComponent` 只执行追击，不关心为什么追。

## 9. 控制模式：PlayerCombatModeComponent

仅挂载于玩家英雄（`UnitCamp.Friend`）。

- `Mode`：`Manual`（手动）/ `Auto`（自动）
- `IsAutoBattle`：属性快捷访问 `Mode == BattleMode.Auto`
- `SetMode(BattleMode)`：切换模式；手动模式下停止移动并重置决策状态。
- `BattleAIComponent.Update` 检查 `IsAutoBattle`，未开启时跳过整个决策链路。

## 11. 维护注意事项

- **不要把所有逻辑塞回 AI 组件**：AI 入口负责"启停控制"，决策层负责"做什么"，移动层负责"怎么移动"。
- **FollowTarget 是客户端同步的关键语义**：`followTargetUnitId / followDistance / followDirectionSign` 的一致性非常重要。
- **冻结恢复依赖 InterruptedCommand**：被中断的移动命令保存在 `BattleMoveComponent.InterruptedCommand`，`FreezeComponent` 自身不持有移动状态。
- **服务端时间统一使用 ServerFrameTime**：`BattleUnitCombatComponent` 所有冷却时间基于 `TimeInfo.Instance.ServerFrameTime()`。

## 12. 一句话总结

**BattleAIComponent 决定是否进入自动链路，BattleActionDecisionComponent 基于技能与配置做行为决策，BattleMoveComponent 执行移动/击退并向客户端同步，BattleUnitCombatComponent 管理冷却，EffectApplyComponent 通过事件系统分发效果，FreezeComponent 管理冻结状态与移动中断恢复，BattleSkillHelper 负责实际的技能判定与效果结算。**
