# 武器标签与修饰器系统设计文档

> 版本: 1.0 | 日期: 2026-04-11
>
> 设计目标: 在现有 SkillConfig → BuffGroupConfig → BuffConfig 链路上，
> 以最小侵入的方式增加**标签分类**和**修饰器修饰**两层抽象，
> 使武器(技能)系统具备高扩展性，支持后续 Synergy、技能进化、属性克制等功能。

---

## 一、问题分析

### 1.1 现有架构

```
UnitCombatConfig (武器载体)
  ├── NormalAttackSkillId  →  SkillConfig →  SkillTargetingConfig + BuffGroupConfig
  └── AutoSkillIds[]       →  SkillConfig →  SkillTargetingConfig + BuffGroupConfig
                                                    ↓
                                            BuffGroupConfig
                                                EffectIds[]
                                                    ↓
                                              BuffConfig[] (Damage / Freeze / Knockback / Heal / Stun)
```

**优势**: 解耦清晰，加新技能/新效果只需加 Excel 行。

### 1.2 瓶颈

| 瓶颈 | 具体表现 | 影响 |
|------|----------|------|
| **无标签** | 技能没有属性分类（火/冰/雷），被动效果无法按标签筛选 | 无法实现 "冰霜附魔 + 冰系技能 = 冻结翻倍" |
| **无修饰器** | BuffConfig.EffectType 是硬编码枚举，新增效果必须改 `BattleSkillHelper.ApplyEffects()` | 每种新行为效果都需要改核心代码 |
| **无升级链** | SkillConfig 扁平结构，无 Level / EvolveFrom 字段 | 无法做 "弹幕 → 散射3发" 的升级 |
| **无运行时参数覆盖** | 效果参数完全由配置决定，运行时无法被外部修改 | 被动/遗物无法临时改变技能参数 |

### 1.3 设计原则

1. **扁平化**: 不引入继承体系，不引入新的 Entity-Component 层级。所有新概念都是**数据层的扩展**（配置字段）+ **事件层的扩展**（新事件/处理器）。
2. **最小侵入**: 不修改现有 `ApplyEffects()` 核心路径，通过事件挂载实现扩展。
3. **配置驱动**: 标签、修饰器全部通过 Luban Excel 配置，新增内容不需要改 C# 代码。
4. **渐进式**: 可以从"只加标签"开始，逐步叠加修饰器能力。

## 二、标签系统 (Tag System)

### 2.1 设计思路

在 `SkillConfig` 和 `BuffConfig` 上各增加一个 `Tags` 字段（`int[]`），用枚举 ID 表示标签。

**标签不是代码中的枚举**，而是 Luban Excel 中的枚举配置（`__enums__.xlsx`），新增标签只需改 Excel，不需改 C# 代码。

### 2.2 Luban 枚举定义

在 `__enums__.xlsx` 中新增枚举表 `SkillTag`:

| 名字 | 值 | 别名 | 注释 |
|------|-----|------|------|
| None | 0 | | 无标签 |
| Fire | 1 | | 火属性 |
| Ice | 2 | | 冰属性 |
| Thunder | 3 | | 雷属性 |
| Physical | 4 | | 物理属性 |
| Poison | 5 | | 毒属性 |
| Melee | 10 | | 近战 |
| Ranged | 11 | | 远程 |
| Projectile | 12 | | 弹道类 |
| AOE | 13 | | 范围类 |
| Heal | 20 | | 治疗类 |
| Summon | 21 | | 召唤类 |
| Buff | 22 | | 增益类 |
| Control | 23 | | 控制类 |
| DOT | 24 | | 持续伤害 |

> **设计要点**: 标签 ID 预留间隔（0-9属性系, 10-19形态系, 20-29功能系），方便后续插入新标签。

### 2.3 配置字段变更

**SkillConfig 新增字段:**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| Tags | `int[]` | 技能标签列表（可多选） |

示例:
- 火球术 (SkillId=101): `Tags = [1, 11, 12]` → 火 + 远程 + 弹道
- 冰霜斩 (SkillId=102): `Tags = [2, 10]` → 冰 + 近战
- 雷暴 (SkillId=103): `Tags = [3, 11, 13]` → 雷 + 远程 + 范围

**BuffConfig 新增字段:**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| Tags | `int[]` | 效果标签列表（可多选） |

### 2.4 使用方式（查询侧）

标签系统的核心价值是**查询侧**——被动效果、Synergy、克制判定等通过标签匹配触发。

不需要专门的组件或系统。任何需要检查标签的地方，直接读配置:

```csharp
// 检查技能是否具有指定标签
public static bool HasTag(this SkillConfig skillConfig, int tag)
{
    if (skillConfig.Tags == null) return false;
    for (int i = 0; i < skillConfig.Tags.Length; i++)
    {
        if (skillConfig.Tags[i] == tag) return true;
    }
    return false;
}
```

## 三、修饰器系统 (Modifier System)

### 3.1 核心思想

修饰器是一个**纯数据结构**，描述"对某个技能效果参数的运行时修改"。

它**不是一个 Component**，不是一个 Entity，而是一段**配置数据**，在效果执行前被读取并叠加到参数上。

### 3.2 修饰器配置表 (ModifierConfig)

新增 Luban 配置表 `ModifierConfig.xlsx`:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| Id | `int` | 修饰器ID |
| Name | `string` | 名称 |
| TargetType | `int` | 修饰对象: 1=Skill, 2=BuffEffect, 3=Projectile |
| ApplyRule | `int` | 应用规则 (见下表) |
| ParamName | `string` | 被修饰的参数名 (见下表) |
| ModifyOp | `int` | 修改方式: 1=Add, 2=Multiply, 3=Override, 4=AddTag |
| Value | `float` | 修改值 |
| TagFilter | `int[]` | 标签过滤: 仅对含有这些标签的技能/效果生效 (空=全部) |
| Priority | `int` | 优先级 (数值小先执行) |
| SourceType | `int` | 来源类型: 1=Passive(被动), 2=Relic(遗物), 3=Buff(增益buff) |
| SourceId | `int` | 来源配置ID |
| DurationMs | `int` | 持续时间(ms), 0=永久(战斗期间) |
| StackType | `int` | 叠加方式: 0=不可叠加, 1=取最大值, 2=累加, 3=累乘 |
| MaxStack | `int` | 最大叠加次数 |
| Desc | `string` | 描述 |

### 3.3 ApplyRule (应用规则)

| 值 | 规则 | 说明 |
|----|------|------|
| 1 | OnCast | 技能施放时生效一次 |
| 2 | OnHit | 每次命中时生效 |
| 3 | OnTick | 每次Buff tick时生效 |
| 4 | Always | 始终生效（被动类） |

### 3.4 ParamName (可修饰参数)

参数名用字符串匹配，对应现有配置中的字段:

| ParamName | 适用的目标类型 | 说明 |
|-----------|---------------|------|
| `DamageRatioAtk` | BuffEffect | 伤害攻击系数 |
| `DamageBaseValue` | BuffEffect | 伤害基础值 |
| `CooldownMs` | Skill | 冷却时间 |
| `CastRange` | Skill (通过TargetingConfig) | 施法距离 |
| `ProjectileSpeed` | Skill / Projectile | 弹道速度 |
| `ProjectilePiercing` | Skill / Projectile | 是否穿透 |
| `ProjectileMaxHitCount` | Skill / Projectile | 最大命中数 |
| `ProjectileMaxDistance` | Skill / Projectile | 最大飞行距离 |
| `DurationMs` | BuffEffect | 效果持续时间 |
| `KnockbackDistance` | BuffEffect | 击退距离 |
| `HealValue` | BuffEffect | 治疗量 |
| `AttackSpeed` | Skill | 攻击速度系数 |

### 3.5 ModifyOp (修改方式)

| 值 | 方式 | 公式 | 示例 |
|----|------|------|------|
| 1 | Add | `param + Value` | 伤害 +50 |
| 2 | Multiply | `param * Value` | 冷却 ×0.8 (缩减20%) |
| 3 | Override | `Value` | 强制覆盖为固定值 |
| 4 | AddTag | 给技能/效果添加一个运行时标签 | 物理技能附加"冰"标签 |

### 3.6 运行时存储: ModifierSlotComponent

修饰器的运行时数据需要一个挂载点。新增一个轻量 Component:

```csharp
// Server/Model/Demo/Battle/ModifierSlotComponent.cs
namespace ET.Server
{
    /// <summary>
    /// 修饰器槽位组件 - 挂载在 BattleUnit 上，管理该单位的所有活跃修饰器。
    /// 扁平设计: 一个 List 存所有修饰器实例，按 Priority 排序。
    /// </summary>
    [ComponentOf(typeof(BattleUnit))]
    public class ModifierSlotComponent : Entity, IAwake, IDestroy
    {
        /// <summary>
        /// 所有活跃修饰器实例
        /// </summary>
        public readonly List<ModifierInstance> Modifiers = new();

        /// <summary>
        /// 标签 → 修饰器列表 的快速查询索引
        /// Key = 标签ID, Value = 含该标签过滤条件的修饰器索引列表
        /// </summary>
        public readonly Dictionary<int, List<int>> TagIndex = new();
    }

    /// <summary>
    /// 修饰器运行时实例 - 纯数据结构，不是 Entity，没有生命周期管理。
    /// 由 ModifierSlotComponent 统一管理添加/移除/过期。
    /// </summary>
    public struct ModifierInstance
    {
        /// <summary>修饰器配置ID</summary>
        public int ModifierId;
        /// <summary>来源类型 (Passive=1, Relic=2, Buff=3)</summary>
        public int SourceType;
        /// <summary>来源配置ID</summary>
        public int SourceId;
        /// <summary>过期时间 (ServerFrameTime), 0=永不过期</summary>
        public long ExpireTime;
        /// <summary>当前叠加层数</summary>
        public int StackCount;
    }
}
```

### 3.7 核心查询方法: GetModifiedParam

这是修饰器系统的**唯一核心API**。在 `BattleSkillHelper` 或 `BuffExecuteEvent_Handler` 中需要读取参数时，调用此方法获取"被修饰后的参数值":

```csharp
// ModifierSlotComponentSystem.cs 中的扩展方法
public static float GetModifiedParam(this ModifierSlotComponent self,
    ModifierConfig modifierConfig, string paramName, float baseValue,
    int[] skillTags = null)
{
    float result = baseValue;

    for (int i = 0; i < self.Modifiers.Count; i++)
    {
        ModifierInstance inst = self.Modifiers[i];
        ModifierConfig modConf = ModifierConfigCategory.Instance.GetOrDefault(inst.ModifierId);
        if (modConf == null) continue;

        // 1. 检查 TargetType 是否匹配
        // 2. 检查 ParamName 是否匹配
        // 3. 检查 TagFilter 是否匹配 (skillTags 为空或包含任一过滤标签)
        // 4. 按 ModifyOp 修改 result
    }

    return result;
}
```

### 3.8 修饰器应用时机 (Hook Points)

修饰器**不修改** `ApplyEffects()` 的核心流程，而是在**读取参数前**调用 `GetModifiedParam()`:

```
现有流程 (不变):
  TryExecuteSkill → ApplyEffects → foreach BuffConfig → CalculateDamage

修饰器介入点 (仅在参数读取时):
  ┌─ CalculateDamage 中:
  │    attack = GetModifiedParam(modifierConfig, "DamageRatioAtk", baseRatioAtk)
  │    damage = CalculateDamage(caster, target, effectConfig, modifiedRatioAtk)
  │
  ├─ TryExecuteSkill 中 (CD修改):
  │    cooldownMs = GetModifiedParam(modifierConfig, "CooldownMs", skillConfig.CooldownMs)
  │    combat.StartSkillCooldown(skillConfig, cooldownMs)
  │
  └─ ProjectileComponent.Launch 中 (弹道修改):
       speed = GetModifiedParam(modifierConfig, "ProjectileSpeed", skillConfig.ProjectileSpeed)
```

**关键设计**: 修饰器系统只在"参数读取"时生效，不改变现有控制流。每处 Hook 都是独立的、可选的——先实现最常用的（伤害系数、CD），再逐步扩展到弹道、范围等。

## 四、事件扩展: SkillHitEvent 增强

### 4.1 现有 SkillHitEvent

```csharp
public struct SkillHitEvent
{
    public BattleUnit Attacker;
    public BattleUnit Target;
    public int SkillId;
    public BuffGroupConfig EffectGroup;
}
```

### 4.2 新增: SkillCastEvent (技能施放事件)

在技能**施放成功**时发布（现有 `M2C_SkillCast` 广播之前），供修饰器、被动效果、Synergy 监听:

```csharp
// 追加到 BattleEvents.cs
/// <summary>
/// 技能施放事件 - 在技能施放成功时触发，早于效果执行。
/// 用于: 被动触发、Synergy判定、修饰器ApplyRule=OnCast。
/// </summary>
public struct SkillCastEvent
{
    public BattleUnit Caster;
    public int SkillId;
    public SkillConfig SkillConfig;
    public List<BattleUnit> Targets;
}
```

### 4.3 新增: SkillHitTagEvent (标签命中事件)

在**每次命中**时发布，携带标签信息，供 Synergy 监听:

```csharp
/// <summary>
/// 技能命中标签事件 - 在技能命中目标时触发，携带命中技能的标签。
/// 用于: 标签Synergy判定（如"冰系技能命中时，30%几率冻结1秒"）。
/// </summary>
public struct SkillHitTagEvent
{
    public BattleUnit Attacker;
    public BattleUnit Target;
    public int SkillId;
    public int[] SkillTags;
    public int Damage;
}
```

### 4.4 事件流时序

```
SkillCastEvent       ← 技能施放成功 (ApplyRule=OnCast 的修饰器在此生效)
  ↓
SelectTargets
  ↓
M2C_SkillCast 广播
  ↓
ApplyEffects / Projectile → 碰撞 → ApplyEffects
  ↓
SkillHitTagEvent     ← 每次命中 (ApplyRule=OnHit 的修饰器在此生效)
  ↓
BuffExecuteEvent     ← Buff tick (ApplyRule=OnTick 的修饰器在此生效)
```

## 五、具体使用场景

### 5.1 场景: "冰霜附魔"被动

**需求**: 被动效果 — 所有攻击附带冰标签，冰系技能伤害+50%。

**实现**:

ModifierConfig 行:

| Id | Name | TargetType | ApplyRule | ParamName | ModifyOp | Value | TagFilter | Priority | SourceType | SourceId | DurationMs | StackType | MaxStack |
|----|------|-----------|-----------|-----------|----------|-------|-----------|----------|-----------|----------|------------|-----------|----------|
| 501 | 冰霜附魔_标签 | 1(Skill) | 4(Always) | Tags | 4(AddTag) | 2 | [] | 10 | 1(Passive) | 301 | 0 | 0 | 1 |
| 502 | 冰霜附魔_增伤 | 2(BuffEffect) | 2(OnHit) | DamageRatioAtk | 1(Add) | 0.5 | [2] | 10 | 1(Passive) | 301 | 0 | 0 | 1 |

行501: 给所有技能添加"冰"标签 (AddTag, Value=2)
行502: 命中时，对有"冰"标签的效果增加50%攻击系数 (TagFilter=[2])

**不需要写任何 C# 代码**，只需配 Excel。

### 5.2 场景: "冷却缩减"遗物

**需求**: 遗物效果 — 所有技能CD减少20%。

**实现**:

| Id | Name | TargetType | ApplyRule | ParamName | ModifyOp | Value | TagFilter | Priority | SourceType | SourceId | DurationMs | StackType | MaxStack |
|----|------|-----------|-----------|-----------|----------|-------|-----------|----------|-----------|----------|------------|-----------|----------|
| 601 | 冷却缩减 | 1(Skill) | 1(OnCast) | CooldownMs | 2(Multiply) | 0.8 | [] | 10 | 2(Relic) | 101 | 0 | 0 | 1 |

**不需要写任何 C# 代码。**

### 5.3 场景: "爆裂尸体"被动

**需求**: 击杀敌人时，在其位置产生一次AOE爆炸。

这个场景**超出了纯修饰器的能力**（需要新增行为逻辑），需要用事件处理器实现:

```csharp
// Server/Hotfix/Demo/Battle/Event/爆裂尸体_Passive_Handler.cs
[Event(SceneType.Battle)]
public class ExplosiveCorpsePassive_Handler : AEvent<Scene, UnitDeadEvent>
{
    protected override async ETTask Run(Scene scene, UnitDeadEvent args)
    {
        // 检查击杀者是否拥有"爆裂尸体"被动
        BattleRoom battleRoom = args.Target.GetParent<BattleRoom>();
        if (battleRoom == null) { await ETTask.CompletedTask; return; }

        BattleUnit killer = battleRoom.GetUnit(args.KillerId);
        if (killer == null) { await ETTask.CompletedTask; return; }

        ModifierSlotComponent modSlot = killer.GetComponent<ModifierSlotComponent>();
        if (!modSlot.HasModifierFromSource(1/*Passive*/, 302/*爆裂尸体ConfigId*/))
        {
            await ETTask.CompletedTask;
            return;
        }

        // 以死者位置为中心，对范围内敌人造成AOE伤害
        // (复用 SkillTimelineComponent 的 RegisterHitBox 逻辑)
        // ...

        await ETTask.CompletedTask;
    }
}
```

**设计思路**: 修饰器处理"参数修改"，事件处理器处理"行为逻辑"。两者组合使用。

### 5.4 场景: 技能进化 "弹幕 → 散射3发"

**需求**: 拿到特定被动后，单体弹道技能变为同时发射3发。

这种**改变行为**的进化需要新增 SkillConfig 条目 + 运行时替换:

```
SkillConfig 1001: 火球术 (单体弹道, ProjectileMaxHitCount=1)
SkillConfig 1002: 火球术_散射 (散射弹道, ProjectileMaxHitCount=3)  ← 进化后版本
```

**实现**: 用一个修饰器 + Hook Point:

| Id | Name | TargetType | ApplyRule | ParamName | ModifyOp | Value | TagFilter | SourceType | SourceId | Desc |
|----|------|-----------|-----------|-----------|----------|-------|-----------|-----------|----------|------|
| 701 | 火球散射 | 1(Skill) | 4(Always) | ProjectileMaxHitCount | 3(Override) | 3 | [1] | 10 | 1(Passive) | 401 | 火属性技能命中数→3 |

> 注意: 进化为"散射"的视觉表现需要客户端配合。修饰器只改了"命中数"参数，客户端需要读取这个参数来决定发射几个弹道。这符合现有架构——客户端也会收到 SkillConfig，读取 ProjectileMaxHitCount 来表现。

## 六、数据流总览

### 6.1 修饰器加载流程

```
战斗开始 (C2M_StartBattle)
  ↓
创建玩家 BattleUnit
  ↓
.AddComponent<ModifierSlotComponent>()
  ↓
加载玩家当前 Build 中的被动/遗物:
  for each passive:
    for each modifier in PassiveConfig.ModifierIds[]:
      ModifierSlotComponent.AddModifier(modifierId, Passive, passiveId)
  for each relic:
    for each modifier in RelicConfig.ModifierIds[]:
      ModifierSlotComponent.AddModifier(modifierId, Relic, relicId)
  ↓
战斗过程中，修饰器始终生效
```

### 6.2 伤害计算流程 (带修饰器)

```
ApplyEffects(caster, target, effectGroupConfig, skillConfig)
  ↓
for each BuffConfig in effectGroupConfig.EffectIds:
  ↓
  if (Damage):
    modifierSlot = caster.GetComponent<ModifierSlotComponent>()
    ↓
    // 读取被修饰后的参数
    float modRatioAtk = modifierSlot.GetModifiedParam(
        null, "DamageRatioAtk", buffConfig.RatioAtk,
        skillTags: skillConfig.Tags)
    float modBaseValue = modifierSlot.GetModifiedParam(
        null, "DamageBaseValue", buffConfig.BaseValue,
        skillTags: skillConfig.Tags)
    ↓
    // 用修饰后的参数计算伤害
    damage = modBaseValue + attack * modRatioAtk - defense * buffConfig.RatioDef
    ↓
    TakeDamage + Broadcast
```

### 6.3 技能施放流程 (带修饰器)

```
TryExecuteSkill(caster, skillId, ...)
  ↓
发布 SkillCastEvent (ApplyRule=OnCast 的修饰器在此生效)
  ↓
SelectTargets → 读取修饰后的 CastRange
  ↓
Broadcast M2C_SkillCast
  ↓
if (Projectile):
  读取修饰后的 ProjectileSpeed / ProjectileMaxHitCount / ProjectilePiercing
  ↓
  SpawnProjectileEvent
else:
  ApplyEffects (命中后发布 SkillHitTagEvent)
  ↓
读取修饰后的 CooldownMs → StartSkillCooldown
```

## 九、与现有架构的兼容性

### 9.1 对现有系统的影响

| 现有系统 | 影响 | 说明 |
|----------|------|------|
| `BuffComponent` / `BuffEntity` | 无影响 | 修饰器是独立系统，不改变 Buff 生命周期 |
| `SkillTimelineComponent` | 无影响 | 命中框注册不受修饰器影响 |
| `ProjectileComponent` | 可选影响 | 弹道参数读取时可用修饰器，但不强制 |
| `BattleActionDecisionComponent` | 无影响 | AI 决策不受修饰器影响（CD 减少后自然更频繁） |
| `ClientMinionAIComponent` | 无影响 | 小兵路径完全不涉及 |
| `BossSyncComponent` | 无影响 | Boss 同步不受修饰器影响 |
| `WaveManagerComponent` | 无影响 | 波次管理不受影响 |
| `C2M_ClientBatchHit` | 可选影响 | 客户端计算伤害时也可读修饰器参数 |

### 9.2 "零修饰器"时的行为

当 `ModifierSlotComponent` 不存在或 `Modifiers` 为空时:

```csharp
// GetModifiedParam 的降级逻辑
if (self == null || self.Modifiers.Count == 0)
    return baseValue;  // 直接返回原始值，零开销
```

**向后兼容**: 旧技能不加 Tags 字段、不挂修饰器，行为与现在完全一致。

### 9.3 性能考量

- `ModifierSlotComponent` 是一个纯 List + Dictionary，查询是 O(N) 遍历 (N=修饰器数量，通常 < 20)
- `GetModifiedParam()` 在伤害计算热路径上调用，但 N 很小，开销可忽略
- **没有**每帧遍历，只在施法/命中时触发
- 标签查询 `HasTag()` 是 O(M) (M=标签数量，通常 < 5)
- 可选优化: 如果未来修饰器数量暴涨，可改为 `Dictionary<(TargetType, ParamName), List<ModifierInstance>>` 索引
