### 3. 创建状态切换辅助类

**位置**：`Unity/Assets/Scripts/Hotfix/Server/Demo/Unit/StateHelper.cs`

✅ **StateHelper.cs** - 核心辅助类
- `ChangeToIdle()` - 切换到待机状态
- `ChangeToMove()` - 切换到移动状态
- `ChangeToAttack(targetId)` - 切换到攻击状态
- `IsIdle()` - 检查是否在待机状态
- `IsMoving()` - 检查是否在移动状态
- `IsAttacking()` - 检查是否在攻击状态
- `GetCurrentStateName()` - 获取当前状态名称（调试用）
- `GetAttackTargetId()` - 获取攻击目标ID

**特性**：
- ✅ 自动处理状态互斥
- ✅ 内置防抖机制（Idle→Move: 200ms, Move→Idle: 100ms）
- ✅ 支持攻击目标切换

## 🎯 核心优势

### 1. 更符合ET风格 ✅
- 使用组件表示状态，而非枚举
- 状态切换通过添加/移除组件实现
- 完全符合ET的ECS架构

### 2. 代码更简洁 ✅
```csharp
// 旧代码：3行
var stateMachine = unit.GetComponent<StateMachineComponent>();
if (stateMachine != null)
    stateMachine.ChangeState(MachineState.Idle);

// 新代码：1行
unit.ChangeToIdle();
```

### 3. 状态互斥自动保证 ✅
- 同一类型的组件只能有一个
- 添加新状态时自动移除旧状态
- 无需手动管理状态转换规则

### 4. 防抖机制内置 ✅
- Idle → Move: 200ms 防抖
- Move → Idle: 100ms 防抖
- 防止状态频繁切换

### 5. 易于扩展 ✅
添加新状态只需3步：
1. 创建新的 StateFlag 组件
2. 创建对应的 System
3. 在 StateHelper 中添加切换方法

## 📝 使用示例

### 业务代码中使用

```csharp
// 1. 切换状态
unit.ChangeToIdle();
unit.ChangeToMove();
unit.ChangeToAttack(targetId);

// 2. 检查状态
if (unit.IsIdle())
{
    // 处于待机状态
}

if (unit.IsMoving())
{
    // 正在移动
}

if (unit.IsAttacking())
{
    // 正在攻击
    long targetId = unit.GetAttackTargetId();
}

// 3. 获取状态名称（调试用）
string stateName = unit.GetCurrentStateName(); // "Idle" / "Move" / "Attack"
Log.Debug($"当前状态: {stateName}");
```

## 🎓 最佳实践

### 1. 状态切换时机
- ✅ 在业务逻辑开始前切换状态
- ✅ 使用 StateHelper 提供的方法
- ❌ 不要直接 AddComponent/RemoveComponent

### 2. 状态判断
- ✅ 使用 `unit.IsIdle()` 等便捷方法
- ✅ 使用 `unit.GetComponent<IdleStateFlag>() != null`
- ❌ 不要缓存状态组件引用

### 3. 防抖配置
- 当前配置：Idle→Move: 200ms, Move→Idle: 100ms
- 可根据实际需求在 StateHelper 中调整

### 4. 日志输出
- 状态进入/退出会自动输出日志
- 格式：`[State] Unit(123) 进入待机状态`
- 可用于调试和问题排查
