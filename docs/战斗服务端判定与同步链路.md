# 战斗服务端判定与同步链路

本文档说明当前服务端战斗链路如何从“自动/手动输入”进入判定，再同步到客户端，重点面向维护者排查“为什么没追击、没放技能、没扣血、没死亡通知”。

## 1. 涉及的关键代码

- `Server/Hotfix/Demo/Battle/BattleAIComponentSystem.cs`
- `Server/Hotfix/Demo/Battle/BattleActionDecisionComponentSystem.cs`
- `Server/Hotfix/Demo/Battle/BattleMoveComponentSystem.cs`
- `Server/Hotfix/Demo/Battle/BattleSkillHelper.cs`
- `Server/Hotfix/Demo/Battle/BattleUnitHelper.cs`
- `Server/Hotfix/Demo/Battle/Handler/C2M_SwitchControlModeHandler.cs`
- `Config/Proto/OuterMessage_C_10001.proto`

## 2. 总体链路

```text
控制模式 / AI 更新
 -> BattleAIComponent.Update
 -> BattleActionDecisionComponent.Update
 -> BattleSkillHelper.TrySelectBestAutoSkillPlan
 -> 追击 or 施法
 -> BattleMoveComponent / BattleSkillHelper.Execute
 -> BattleUnitHelper.BroadcastXXX
 -> 客户端 Handler 收到消息并表现
```

## 3. 自动战斗判定链路

### 3.1 自动战斗入口

入口代码：`BattleAIComponentSystem.Update`

处理逻辑：

1. 取当前 `BattleUnit`。
2. 如果单位为空或已死亡，直接返回。
3. 如果是友方单位，则检查 `PlayerCombatModeComponent.IsAutoBattle`。
4. 若当前不是自动战斗：
   - `BattleMoveComponent.StopMove()`
   - `BattleActionDecisionComponent.Reset()`
5. 若允许自动战斗，则进入 `BattleActionDecisionComponent.Update()`。

### 3.2 生成自动施法方案

入口代码：`BattleActionDecisionComponentSystem.Update`

内部通过 `BattleSkillHelper.TrySelectBestAutoSkillPlan` 生成 `AutoCastPlan`，其中包含：

- `SkillId`
- `Target`
- `TargetingConfig`
- `DesiredCastPosition`
- `DesiredCastDistance`
- `RequiredMoveDistance`

如果没有可行方案：

- 状态切回 `Idle`
- 停止移动
- 当前目标清空

### 3.3 追击还是攻击

`BattleActionDecisionComponentSystem.Update` 根据方案做三选一：

#### 分支 A：已在施法范围内

- `BattleMoveComponent.StopMove()`
- 状态设为 `Attacking`
- 调用 `TryAttack()`

#### 分支 B：目标切换但位移需求很小

- 进入 `Attacking` 状态，但不立即追击
- 用于抑制切目标时的来回抖动

#### 分支 C：仍需靠近目标

- 状态设为 `Chasing`
- 调用 `BattleMoveComponent.FollowTarget(target, desiredCastDistance)`

## 4. 移动同步链路

### 4.1 服务端如何决定广播移动

`BattleMoveComponentSystem` 只在必要时广播移动命令，而不是每帧广播位置。

#### 固定点移动

`StartMove(targetPosition)` 会在以下场景广播：

- 尚未发送过移动命令
- 上一次是跟随命令
- 目标点变化超过阈值 `MoveCommandTargetThreshold`

广播消息：`M2C_BattleUnitMoveCommand`

#### 跟随目标移动

`FollowTarget(followTarget, followDistance)` 会在以下场景广播：

- 尚未发送过移动命令
- 跟随目标变化
- 跟随距离变化超过阈值
- 跟随站位方向变化

消息字段意义：

```protobuf
message M2C_BattleUnitMoveCommand
{
    int64 battleId = 1;
    int64 unitId = 2;
    float3 targetPosition = 3;
    float moveSpeed = 4;
    bool isMoving = 5;
    int64 followTargetUnitId = 6;
    float followDistance = 7;
    float followDirectionSign = 8;
}
```

关键点：

- `followTargetUnitId = 0` 表示普通点位移动。
- 非 0 表示客户端应按“跟随目标”语义解释该移动命令。

#### 停止移动

`StopMove()` 在已有激活移动命令时，会广播一次：

- `isMoving = false`
- 目标点为当前停留位置

### 4.2 服务端位置推进

`BattleMoveComponentSystem.Update()` 负责服务端权威位置推进：

- 若为跟随目标，先重算最新目标点
- 根据 `deltaTime` 和 `moveSpeed` 计算位移
- 到达后自动 `StopMove()`
- 若跟随目标死亡 / 消失，立即停止

因此：

- 移动路径和停止点以服务端为准。
- 客户端主要根据命令播放追击与停走表现。

## 5. 技能判定与伤害链路

### 5.1 技能选择

`BattleSkillHelper.TrySelectBestAutoSkillPlan` 通过以下配置驱动自动战斗：

- `UnitCombatConfig`：该单位可自动施放哪些技能
- `AutoBattleStrategyConfig`：先选技能还是先选目标、是否优先 Boss / 精英 / 当前目标
- `SkillConfig`：冷却、优先级、是否启用
- `SkillTargetingConfig`：施法距离、目标关系、目标数量、排序规则

### 5.2 可施法校验

`BattleSkillHelper.TryExecuteSkill` 主要校验：

1. 施法者是否存在且未死亡
2. BattleRoom 是否存在
3. `BattleUnitCombatComponent` 是否存在
4. `SkillConfig` / `SkillTargetingConfig` / `SkillCastCheckConfig` / `SkillEffectGroupConfig` 是否完整
5. 是否允许攻击
6. 是否在冷却中
7. 是否受自动战斗限制
8. 是否能选出合法目标
9. 是否在施法范围内

### 5.3 目标筛选

`SelectTargets` 支持当前实现中的几类目标选择：

- 锁定显式目标
- 最近敌人
- 范围内所有敌人

筛选后还会：

- 按 `SortRule` 排序
- 按 `MaxTargetCount` 截断

### 5.4 伤害结算

`ApplyEffects` 会遍历 `SkillEffectGroupConfig.EffectIds`，目前文档确认到的已实现效果类型为：

- `EffectTypeDamage = 1`

伤害公式由 `SkillEffectConfig.FormulaType` 决定，当前已见到：

- `Attack - Defense`
- 固定值

目标受伤后执行：

1. `target.TakeDamage(damage)`
2. `BattleUnitHelper.BroadcastDamage(...)`
3. 若此前存活且现在死亡，再执行 `BattleUnitHelper.BroadcastUnitDead(...)`

## 6. 服务端到客户端的消息广播

广播集中在 `BattleUnitHelper`。

### 6.1 施法消息

```protobuf
message M2C_SkillCast
{
    int64 casterId = 1;
    int32 skillId = 2;
    int64 targetId = 3;
    float3 targetPos = 4;
}
```

发送时机：`TryExecuteSkill` 选出主目标后，先广播施法，再执行效果。

### 6.2 伤害消息

```protobuf
message M2C_Damage
{
    int64 attackerId = 1;
    int64 targetId = 2;
    int32 damage = 3;
    bool isCrit = 4;
    int32 targetCurrentHp = 5;
    int32 targetMaxHp = 6;
    bool targetDead = 7;
    int32 damageType = 8;
}
```

发送时机：每次应用伤害效果后立即广播。

### 6.3 死亡消息

```protobuf
message M2C_UnitDead
{
    int64 unitId = 1;
    int64 killerId = 2;
}
```

发送时机：单位本次受击前还活着，受击后变为死亡。

## 7. 控制模式切换链路

控制模式消息由 `C2M_SwitchControlModeHandler` 处理。

### 7.1 服务端处理流程

1. 从 `BattleRoomManagerComponent` 找到玩家所在战斗房间。
2. 找到友方 `BattleUnit`。
3. 获取或创建 `PlayerCombatModeComponent`。
4. 调用 `SetMode(newMode)`。
5. 返回 `M2C_SwitchControlMode` 响应。
6. 向房间内广播 `M2C_ControlModeChanged`。

### 7.2 对自动战斗的实际影响

`PlayerCombatModeComponent.SetMode` 会影响：

- 是否允许 `BattleAIComponent` 继续驱动自动战斗
- 是否停止当前移动
- 是否重置当前判定状态

所以控制模式本质上是“自动战斗总开关”，而不是单纯 UI 状态。

## 8. 常见问题排查

### 8.1 单位一直不追击

优先检查：

1. `BattleAIComponent.Update` 是否执行
2. 玩家是否处于自动模式
3. `TrySelectBestAutoSkillPlan` 是否返回 false
4. `BattleMoveComponent.FollowTarget` 是否被调用
5. 是否广播了 `M2C_BattleUnitMoveCommand`

### 8.2 技能没放出来

优先检查：

1. `TryExecuteSkill` 是否进入
2. `SkillConfig` 是否启用
3. `CastCheckConfig` 是否拦截了攻击 / 冷却 / 自动模式
4. 目标是否在 `SkillTargetingConfig` 允许范围内
5. 是否广播了 `M2C_SkillCast`

### 8.3 客户端没扣血

优先检查：

1. `ApplyEffects` 是否执行到伤害效果
2. `target.TakeDamage` 后 HP 是否变化
3. `BattleUnitHelper.BroadcastDamage` 是否发送
4. 客户端是否收到了 `M2C_Damage`

### 8.4 死亡没同步

优先检查：

1. 伤害前 `wasAlive` 是否为 true
2. 伤害后 `target.IsDead` 是否为 true
3. `BroadcastUnitDead` 是否执行
4. 客户端 `M2C_UnitDeadHandler` 是否命中

## 9. 维护建议

- 新增技能效果时，优先从 `BattleSkillHelper.ApplyEffects` 开始扩展，而不是散落到多处处理器。
- 新增战斗同步消息时，优先复用 `BattleUnitHelper.BroadcastToPlayers` 的房间广播模型。
- 调整自动战斗行为时，不要直接绕开 `AutoCastPlan`，否则容易让“选技能 / 追击 / 攻击”三段逻辑失衡。

## 10. 一句话总结

当前服务端战斗链路是：**AI 决定是否进入自动战斗，决策层产出技能驱动的 AutoCastPlan，移动层负责追击与停走，技能层完成校验和结算，最终通过 BattleUnitHelper 把施法、伤害、死亡、移动命令同步给客户端。**
