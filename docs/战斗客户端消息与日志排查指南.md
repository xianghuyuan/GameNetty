# 战斗客户端消息与日志排查指南

本文档整理当前客户端处理战斗同步消息的位置、日志格式与排查方法，重点覆盖移动、施法、伤害、死亡、控制模式 HUD 以及表现反馈。

## 1. 关键代码位置

### 消息处理

- `Unity/Assets/GameScripts/HotFix/GameLogic/Module/Battle/Handler/M2C_BattleUnitMoveCommandHandler.cs`
- `Unity/Assets/GameScripts/HotFix/GameLogic/Module/Battle/Handler/M2C_SkillCastHandler.cs`
- `Unity/Assets/GameScripts/HotFix/GameLogic/Module/Battle/Handler/M2C_DamageHandler.cs`
- `Unity/Assets/GameScripts/HotFix/GameLogic/Module/Battle/Handler/M2C_UnitDeadHandler.cs`

- `Unity/Assets/GameScripts/HotFix/GameLogic/Module/Battle/BattleMessageLogHelper.cs`
- `Unity/Assets/GameScripts/HotFix/GameLogic/Module/Battle/UI/BattleUIHelper.cs`
- `Unity/Assets/GameScripts/HotFix/GameLogic/UI/BattleMainUI/BattleHUDWindow.cs`

## 2. 客户端处理总览

```text
收到服务端战斗消息
 -> 对应 MessageHandler
 -> 输出 BattleMessageLogHelper 注释日志（部分消息）
 -> 查找当前 Battle / BattleUnit
 -> 更新 BattleUnit 数据或 BattleMoveComponent
 -> 触发表现层 / UI 反馈
```

当前客户端不是纯日志记录，而是“消息即驱动”：

- 移动消息驱动 BattleMoveComponent
- 伤害消息驱动 HP 同步和受击反馈
- 死亡消息驱动 BattleUnit 死亡事件
- 控制模式消息驱动 HUD 标签

## 3. 已接入的关键消息

## 3.1 `M2C_BattleUnitMoveCommand`

处理器：`M2C_BattleUnitMoveCommandHandler`

处理流程：

1. 取 `BattleComponent.GetCurrentBattle()`。
2. 先输出 `BattleMessageLogHelper.FormatMoveCommand` 日志。
3. 校验 `battleId` 是否匹配当前战斗。
4. 找到对应 `BattleUnit`。
5. 获取或创建 `BattleMoveComponent`。
6. 根据 `message.isMoving`：
   - true：`ApplyMoveCommand(...)`
   - false：`StopMove(...)`

日志样例：

```text
[收到消息注释] M2C_BattleUnitMoveCommand -> 玩家 开始跟随 敌人1，保持距离 1.20，站位方向 目标左侧，速度 3.50，当前目标点 (4.20, 0.00, 0.00)
```

排查重点：

- 如果有日志但单位不动，优先看客户端 `BattleMoveComponent` 的应用逻辑。
- 如果日志显示“未找到战斗单位”，说明服务端 UnitId 与当前 Battle 本地实体未对齐。

## 3.2 `M2C_SkillCast`

处理器：`M2C_SkillCastHandler`

处理流程：

1. 获取当前 Battle。
2. 输出 `FormatSkillCast` 注释日志。
3. 找到施法者 `BattleUnit`。
4. 停止施法者当前移动表现。
5. 通过 `BattleUnitViewComponent.PlayAttackFeedback(root, casterId)` 播放攻击反馈。

日志样例：

```text
[收到消息注释] M2C_SkillCast -> 玩家 释放技能 1001，目标 敌人1，目标位置 (5.00, 0.00, 0.00)
```

排查重点：

- 有日志无动作反馈：优先检查 `BattleUnitViewComponent` 是否存在。
- 无日志：优先回到服务端检查是否广播了 `M2C_SkillCast`。

## 3.3 `M2C_Damage`

处理器：`M2C_DamageHandler`

处理流程：

1. 获取当前 Battle。
2. 输出 `FormatDamage` 注释日志。
3. 找到目标 `BattleUnit`。
4. 同步本地数值：
   - `SetNumeric(NumericType.MaxHp, message.targetMaxHp)`
   - `SetNumeric(NumericType.Hp, message.targetCurrentHp)`
5. 调用 `BattleUnitViewComponent.PlayHitFeedback(root, targetId, damage)` 播放受击反馈与数字飘字。

日志样例：

```text
[收到消息注释] M2C_Damage -> 玩家 对 敌人1 造成 35 点技能伤害，目标血量 65/100，死亡=False
```

排查重点：

- 有日志但血条不变：检查 HUD 是否订阅到了数值变化。
- 有日志但没飘字 / 没受击：检查 `PlayHitFeedback` 调用链。

## 3.4 `M2C_UnitDead`

处理器：`M2C_UnitDeadHandler`

处理流程：

1. 获取当前 Battle。
2. 输出 `FormatUnitDead` 注释日志。
3. 找到目标 `BattleUnit`。
4. 若单位未死亡，则：
   - `battleUnit.IsDead = true`
   - `EventSystem.Instance.Publish(root, new BattleUnitDead { BattleUnit = battleUnit })`

日志样例：

```text
[收到消息注释] M2C_UnitDead -> 敌人1 已死亡，击杀者 玩家
```

后续联动：

- `BattleUnitDead_UI` 会关闭该单位的 HUD 组件。
- 其他表现事件也可继续订阅 `BattleUnitDead`。

## 3.5 `M2C_ControlModeChanged`

处理器：`M2C_ControlModeChangedHandler`

处理逻辑较轻：

- 直接调用 `BattleUIHelper.OnControlModeChanged(message.UnitId, message.NewMode)`
- 最终更新 `BattleHUDWindow` 中的控制模式文案

当前 HUD 文案：

- `1 => 自动`
- `0 => 手动`

显示格式：

```text
当前战斗模式：自动
当前战斗模式：手动
```

## 4. BattleMessageLogHelper 的作用

`BattleMessageLogHelper` 是当前战斗排查最直接的入口，负责把协议消息翻译成维护者更容易阅读的中文注释日志。

### 已提供的格式化方法

- `FormatMoveCommand`
- `FormatSkillCast`
- `FormatDamage`
- `FormatUnitDead`

### 标签解析逻辑

`GetUnitLabel` 会把单位 ID 转成更易读的标签：

- 友方：`玩家`
- 敌方：按当前战斗内排序显示为 `敌人1 / 敌人2 / ...`
- 找不到实体：回退为 `单位{unitId}`

这对排查有两个价值：

1. 快速分辨是哪一类单位出了问题。
2. 当日志退回 `单位{unitId}` 时，往往意味着当前本地 Battle 实体不完整或消息时序不一致。

## 5. HUD 与表现层联动

## 5.1 控制模式 HUD

代码位置：`BattleHUDWindow`

要点：

- 界面创建时通过 `BattleUIHelper.BindHUD(this)` 绑定。
- `SetControlMode(int mode)` 会刷新 `m_tmpControlMode` 文案。
- 当前默认逻辑中，友方单位 UI 初次创建且模式未设置时，会先给一个 `1`（自动）的默认显示。

维护注意：

- 如果服务端默认模式未来改动，这里的默认显示逻辑也要同步评估。

## 5.2 单位数值与死亡 UI

`BattleUIHelper` 目前负责：

- 创建单位 HUD
- 数值变化时刷新 HUD
- 控制模式变化时刷新 HUD
- 单位死亡时移除 HUD
- 战斗结束时清空 HUD

事件绑定如下：

- `BattleUnitCreated_UI`
- `BattleUnitNumericChange_UI`
- `BattleUnitDead_UI`
- `BattleEnd_UI`

## 5.3 战斗反馈

当前消息处理器已经明确接入的反馈包括：

- `PlayAttackFeedback`：收到 `M2C_SkillCast` 时播放
- `PlayHitFeedback`：收到 `M2C_Damage` 时播放

因此排查“客户端没反馈”时，要先区分：

- 是否没收到消息
- 是否收到但没找到目标单位
- 是否收到也找到单位，但 ViewComponent 缺失

## 6. 推荐排查顺序

### 6.1 单位不移动 / 追击异常

1. 看客户端是否有 `M2C_BattleUnitMoveCommand` 注释日志。
2. 看日志是普通移动、跟随移动还是停止移动。
3. 看是否提示“未找到战斗单位”。
4. 继续检查 BattleMoveComponent 本地执行。

### 6.2 技能没表现

1. 看是否有 `M2C_SkillCast` 日志。
2. 看日志中的施法者 / 目标是否能正常解析成人类可读标签。
3. 看 `BattleUnitViewComponent` 是否存在。
4. 看施法时是否被移动逻辑打断。

### 6.3 伤害数字没显示

1. 看是否有 `M2C_Damage` 日志。
2. 看目标当前血量是否已经同步变化。
3. 看 `PlayHitFeedback` 是否执行。
4. 若血量变化但 UI 不变，再查 HUD 数值刷新链路。

### 6.4 死亡后 HUD 没消失

1. 看是否有 `M2C_UnitDead` 日志。
2. 看 `battleUnit.IsDead` 是否被置为 true。
3. 看 `BattleUnitDead` 事件是否发布。
4. 看 `BattleUnitDead_UI` 是否成功关闭对应单位 UI。

### 6.5 控制模式文字不对

1. 看服务端是否广播 `M2C_ControlModeChanged`。
2. 看 `M2C_ControlModeChangedHandler` 是否执行。
3. 看 `_hudWindow` 是否已绑定。
4. 看 `BattleHUDWindow.RefreshControlModeLabel()` 最终文案。

## 7. 日志使用建议

- 当前“收到消息注释”日志非常适合作为联调和回归测试的首层证据。
- 排查时优先看 `M2C_SkillCast -> M2C_Damage -> M2C_UnitDead` 是否形成完整链。
- 若消息日志都正常但表现异常，问题大概率在客户端实体状态或 View / UI 层。

## 8. 一句话总结

当前客户端战斗排查的核心方法是：**先看 BattleMessageLogHelper 输出的消息注释，再定位对应 Handler 是否更新了 BattleUnit、本地移动组件、表现组件和 HUD。**
