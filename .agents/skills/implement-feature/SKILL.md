# 实现新功能模块

用户需求：$ARGUMENTS

## 第零步：二次确认

先判断用户意图是否匹配本流程（实现完整新系统/模块）。如果匹配，用 AskUserQuestion 确认：

> 检测到你要实现「xxx」，是否启动完整实现流程？流程包含：方案设计 → 配置表导出 → 协议导出 → 服务端代码 → 客户端代码。

用户选择"是"后进入第一步。选择"否"则按普通方式处理。

## 第一步：解析需求 + 查阅文档

1. 分析需求，明确功能范围
2. 读取 `docs/` 目录下所有相关规范文档作为设计依据（如 `docs/gameplay.md`）
3. 读取 `AGENTS.md` 了解框架约定和目录结构
4. 查找项目中已有的类似实现作为代码参考
5. 如文档中没有覆盖该功能的规范，提示用户需要先补充文档

## 第二步：生成方案文档

将方案写入 `.Codex/plans/<功能名>.md`，必须包含以下章节：

- **需求概述** — 功能简要描述
- **配置表设计** — 表名、字段（名/类型/说明）、`__tables__.xlsx` 注册信息
- **协议设计** — 每条消息的名称（C2M_/M2C_）、字段（名/类型/说明）
- **服务端代码** — 新建文件清单（Model层 + Hotfix层）、新增组件表（组件名/挂载实体/关键字段/关键方法）
- **客户端代码** — 新建文件清单、新增组件表
- **文件清单** — 所有新建和修改的文件列表

## 第三步：用户确认

**必须暂停**。用 AskUserQuestion 展示方案摘要，等用户确认后再继续。

## 第四步：调度子代理

用户确认后，依次用 Agent 工具调用4个子代理。每次调用时传入方案文档路径，让子代理读取对应章节执行：

1. **config** — 读「配置表设计」→ 创建 Excel + Luban 导出
2. **proto** — 读「协议设计」→ 创建 Proto + 更新双端代码 + 编译验证
3. **server** — 读「服务端代码」→ 创建组件/系统/Handler + 编译验证
4. **client** — 读「客户端代码」→ 创建组件/系统/Handler

每个子代理完成后检查其报告，如有错误则修正后继续下一步。

## 第五步：最终汇总

全部完成后，汇总4个子代理的报告，输出：
- 创建/修改的文件列表
- 编译状态
- 注意事项（如需要手动填写的 Excel 数据等）
