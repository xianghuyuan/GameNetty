# Plans 目录规范

`.Codex/plans/` 用于存放具体任务、功能、系统或资源需求的落地文档。这里的文件不是通用模板，而是某一次真实研发工作的方案、规格、清单、状态或验收记录。

## 与其他目录的关系

```text
.Codex/workflows/   # 通用流程：这类事情通常怎么做
.Codex/templates/   # 文档模板：某类文档应该怎么写
.Codex/plans/       # 具体实例：这一次任务的方案、规格、清单和记录
```

新需求进入时，优先从 `.Codex/workflows/requirement-intake-workflow.md` 开始。它负责把一句目标整理成 Feature Spec 草稿，并判断后续需要进入哪些 workflow 或 skill。

## 推荐文档类型

### Feature Spec

用于描述一个功能或系统的目标、范围、规则、影响面和验收标准。

推荐命名：

```text
<功能名>-feature-spec.md
```

### Prefab Spec

用于描述一个 UI Prefab 的层级、组件、绑定字段、资源需求和验收标准。

推荐命名：

```text
<Prefab名>-prefab-spec.md
```

### Harness

用于记录一次具体任务的状态、检查点、验证结果、阻塞项和交付信息。

推荐命名：

```text
<任务名>-harness.md
```

### Checklist

用于拆分程序、美术、配置、测试等执行清单。

推荐命名：

```text
<功能名>-程序实现清单.md
<功能名>-美术需求清单.md
<功能名>-验收清单.md
```

### Supplement

用于保留已降级、已废弃或只作为补充说明的方案。

推荐命名：

```text
<主题>-补充说明.md
<主题>-已降级方案.md
```

## 推荐文档头

每个 plan 建议在标题后保留一个统一的元信息块：

```text
## 文档元信息

- 文档类型：
- 状态：
- 使用 Workflow：
- 来源：
- 上游文档：
- 下游文档：
- 最近整理：
```

## 状态枚举

推荐使用以下状态：

```text
草稿
候选方案
主方案
实现拆分
资源需求
验证中
已完成
已降级
已废弃
补充说明
```

## 使用原则

- 先用模板生成新文档，再在 `plans/` 中落成具体实例。
- 主方案负责方向，不混入过多执行细节。
- 程序、美术、Prefab、配置、验收清单可以拆成独立 plan。
- 已降级方案不直接删除，标注状态和当前主方案链接。
- 当一个 plan 已经变成稳定规范，可以再上移沉淀到 `workflows/` 或 `templates/`。
