## 1. 目标

第一版目标不是做完整装备系统，而是先跑通“角色特性 + 局内肉鸽选择 + Emitter/Buff 内部执行模型”的闭环：

1. 击杀怪物累计质量权重
2. 达到阈值或关键怪保底时触发单点候选成长项
3. 战斗完全暂停
4. 玩家执行接受 / 替换 / 保留
5. 旧成长物件回收为局内钱币
6. 新成长结果在程序层改写 `Emitter`、`Buff`、`Modifier` 或角色特性放大参数

## 2. 已确认方案

- 不走地图拾取
- 候选成长项单点出现
- 出现时战斗完全暂停
- 玩家层不暴露 Emitter / Buff / Modifier 分类
- 当前实现可以保留内部分类，方便第一版落地
- HUD 只展示少量代表性成长结果，不以传统装备栏表达为目标
- 候选触发机制：`质量权重累积 + 关键怪保底`
- 触发后：`扣阈值并保留溢出`
- 存在最短触发间隔
- 候选类型第一版可按内部分类或成长来源做权重
- 若保留内部展示位，空位阶段可以强制填入
- 首次填入不做品质保底
- 使用统一随机池
- 旧物件替换后转为局内钱币
- 钱币第一版只累计，不消费
- 候选界面需要展示当前路线与新候选对比
- 第一版成长效果只做数值型

## 3. 配置层

### 3.1 BattleDeviceConfig

用途：定义局内成长项。

说明：

- `BattleDevice` 是第一版内部工程名
- 玩家可见名应使用武器、祝福、遗珍、奇物、战利品等题材化表达
- `Type` 可以继续保留内部分类，但不应直接暴露给美术和 UI

建议字段：

- `Id`
- `Name`：玩家可见名
- `InternalType`：Emitter / Buff / Modifier / Trait
- `Quality`
- `ParamTypeA`
- `ValueA`
- `ParamTypeB`
- `ValueB`
- `RecycleCoin`
- `Desc`

### 3.2 BattleDeviceDropConfig

用途：定义候选池。

建议字段：

- `Id`
- `DeviceId`
- `Weight`
- `Type`

### 3.3 MonsterDropWeightConfig

用途：定义怪物质量权重与保底触发能力。

建议字段：

- `MonsterConfigId`
- `DropWeight`
- `GuaranteeTrigger`

### 3.4 EmitterSocketTemplateConfig

第一版不建议把插槽模板作为玩家层主规则。

若程序实现需要内部展示位或调试位，可先不独立配表，直接在配置或常量里定义默认容量。

预留字段建议：

- `Id`
- `MaxVisibleItemCount`
- `MaxInternalEmitterModifierCount`
- `MaxInternalBuffModifierCount`

## 5. 运行时数据结构

### 5.1 BattleDeviceRuntimeData

用途：保存当前局内成长项实例状态。

建议字段：

- `ConfigId`
- `InternalType`
- `Quality`
- `VisibleIndex`

### 5.2 BattleDeviceModifierContext

用途：汇总当前局内成长项带来的运行时修正。

建议分组：

- Emitter 修正
  - `DamagePct`
  - `FireRatePct`
  - `ProjectileCount`
- Buff 修正
  - `PowerPct`
  - `DurationPct`
  - `RangePct`
- Modifier 修正
  - `DamageByBuffCountPct`
  - `FireRateByBuffCountPct`
  - `BuffPowerByFireRatePct`
  - `TraitBonusPct`

### 5.3 BattleDeviceComponent

建议挂载在玩家 `BattleUnit` 上。

职责：

- 维护当前局内成长项列表
- 维护少量 HUD 代表性展示项
- 维护当前局内钱币
- 维护当前累计质量权重
- 维护最近一次触发时间
- 维护当前是否处于候选处理中
- 维护当前修正汇总上下文

建议字段：

- `ActiveDevices`
- `VisibleDevices`
- `CurrentCoin`
- `CurrentDropWeight`
- `LastTriggerTime`
- `IsCandidatePending`
- `ModifierContext`

## 6. 服务端核心流程

### 6.1 初始化

进入战斗时：

1. 给玩家 `BattleUnit` 挂 `BattleDeviceComponent`
2. 初始化局内成长列表为空
3. 初始化局内钱币为 0
4. 初始化累计权重为 0
5. 初始化修正上下文

### 6.2 怪物死亡累计

怪物死亡时：

1. 读取怪物质量权重
2. 累加到玩家 `BattleDeviceComponent.CurrentDropWeight`
3. 若是精英/Boss，则额外走保底触发逻辑
4. 判断是否达到触发阈值
5. 判断是否满足最短触发间隔
6. 判断当前是否已有候选处理中

### 6.3 候选触发

达到条件时：

1. 扣除阈值
2. 保留溢出权重
3. 生成单个候选成长项
4. 标记 `IsCandidatePending = true`
5. 暂停当前战斗流程
6. 下发候选消息给客户端

### 6.4 候选成长项生成

生成规则：

1. 根据当前角色特性、已有成长项和候选池权重抽取 1 个候选
2. 第一版可先使用统一随机池
3. 若需要保证起手完整度，可对武器/祝福/遗珍类候选做轻量权重修正
4. 第一版不做阶段池、不做高级池

### 6.5 玩家选择处理

#### 起手补齐阶段

- 若当前局缺少基础能力，可强制接受
- 不表现为空位装备，而表现为获得起手成长

#### 正常阶段

- 玩家可选择接受
- 玩家可选择保留当前路线
- 若接受会替换某个代表性旧物件，旧物件转为局内钱币

### 6.6 替换处理

若选择接受：

1. 判断该成长是新增、叠加还是替换代表性旧物件
2. 若存在被替换旧物件，则按 `RecycleCoin` 转为局内钱币
3. 新成长项写入 `ActiveDevices`
4. 重算修正上下文
5. 应用到 `Emitter`、`Buff`、`Modifier` 或角色特性运行时参数

若选择保留：

1. 不改当前成长项
2. 不增加钱币
3. 直接恢复战斗

### 6.7 恢复战斗

选择结束后：

1. 清理候选处理中状态
2. 同步当前成长项、代表性展示项与钱币
3. 恢复战斗

## 7. 参数生效方式

### 7.1 武器/能力类成长

第一版只影响：

- Emitter 伤害
- Emitter 触发频率
- Emitter 触发数量 / 目标数量

建议通过 Emitter 运行时参数上下文实现，不直接改底层模板配置。

### 7.2 祝福类成长

第一版只影响：

- Buff 强度
- Buff 持续时间
- Buff 作用范围

建议通过 Buff 运行时参数上下文实现。

### 7.3 遗珍/奇物类成长

第一版只做桥接型数值或轻量规则，不做复杂事件触发。

建议支持：

- Buff 数量影响 Emitter 伤害
- Buff 数量影响 Emitter 触发频率
- Emitter 触发频率影响 Buff 强度
- 角色特性对某类成长额外增幅

### 7.4 NumericComponent 使用边界

建议：

- 基础属性仍走 `NumericComponent`
- 局内成长修正优先走 Emitter/Buff/Modifier 运行时上下文
- 第一版不要把局内成长逻辑全部塞进 `NumericComponent`

客户端镜像当前局内成长状态：

- 当前成长项列表
- 当前代表性展示项
- 当前局内钱币

### 9.2 候选弹窗逻辑

需要支持：

- 接收候选消息

- 展示当前路线与新候选对比

### 9.3 HUD 展示

需要支持：

- 少量代表性成长物件图标
- 钱币数显示

### 9.4 调试面板

建议开发阶段提供：

- 当前累计权重
- 当前触发阈值
- 当前最短触发间隔剩余
- 当前成长项列表
- 当前代表性展示项
- 当前局内钱币
- 当前修正上下文

## 10. 测试清单

至少验证以下内容：

1. 普通怪击杀能正常累计质量权重
2. 精英/Boss 保底触发正常
3. 达阈值后扣除阈值并保留溢出
4. 最短触发间隔生效
5. 候选处理中不会重复触发
6. 起手补齐阶段能强制接受基础成长
7. 正常阶段可接受/可保留
8. 替换代表性旧物件后能正确转为钱币
9. 武器/能力类成长能立即作用到 Emitter
10. 祝福类成长能立即作用到 Buff
11. 遗珍/奇物类成长能通过 Modifier 生效
12. 暂停与恢复不破坏战斗流程
13. 重连后当前成长项、代表性展示项、钱币、权重不丢失

## 12. 第一版结论

第一版重点不是复杂策略，而是验证以下体验是否成立：

> 玩家在推图过程中，通过单点候选、暂停选择和即时生效，能够明显感知“局内构筑正在逐渐成型”。
