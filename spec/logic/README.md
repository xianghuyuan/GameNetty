# 功能逻辑文档

本目录存放功能模块的逻辑规格，按功能领域形成层级。文档关注玩法规则、系统设计、协议/配置、服务端底层逻辑、客户端底层逻辑、同步边界、HUD/界面状态逻辑和验收标准。

GameNetty 同时包含 .NET 8.0/ET 服务端和 Unity/TEngine 客户端。每个功能逻辑 spec 不能只写玩法描述，必须分别提炼服务端和客户端的底层逻辑，并明确两端通过协议、配置和事件如何衔接。

## 目录层级

```text
spec/logic/
  core/          # 核心玩法、全局规则、跨系统基础能力
  battle/        # 战斗域
    ai/          # 战斗 AI、自动战斗、目标选择、性能约束
    view/        # 战斗单位表现层、逻辑到表现的映射
    hud/         # 战斗内 HUD 状态、界面交互逻辑和 Widget 绑定
    vehicle/     # 战斗中的载具、载具 Buff、载具成长和战斗效果逻辑
```

新增功能文档应优先放入已有领域目录。只有当功能无法归入现有层级，且会持续承载多个子模块时，才新增一级或二级目录。

## 编写约束

### 必须区分服务端与客户端

功能逻辑文档必须明确以下边界：

- **服务端负责什么**：权威数据、状态校验、战斗判定、持久化、配置读取、计时器、Actor/Location 路由、消息处理。
- **客户端负责什么**：输入、表现、局部预测、本地 AI/临时实体、UI 刷新、资源/Prefab 绑定、事件转发表现。
- **两端如何同步**：C2M/M2C 消息、配置表字段、事件发布、同步频率、客户端等待服务端确认还是本地先行。
- **权威归属**：哪些数据以服务端为准，哪些可以客户端权威，哪些只是表现态。

### 服务端底层逻辑必须说明

每个涉及服务端的功能模块都应说明：

- 所属场景和实体：例如 Map Scene、BattleRoom、Unit、BattleUnit。
- 组件设计：新增或复用哪些 Component，挂载在哪个 Entity 上，生命周期如何创建和销毁。
- 系统入口：消息 Handler、Timer、Event、EntitySystem 的触发点。
- 数据来源：持久化数据、NumericComponent、Luban 配置、运行时状态。
- 权威规则：服务端需要校验什么、拒绝什么、如何防止客户端伪造。
- 状态变化：关键状态如何写入、回滚、广播和清理。
- 性能约束：是否会在 1000+ 小怪、20ms/100ms tick、广播同步等场景下产生压力。

### 客户端底层逻辑必须说明

每个涉及客户端的功能模块都应说明：

- 所属模块：`GameLogic` 中的 Module/UI/Handler/Event/View 位置。
- 组件设计：客户端 Entity/Component 如何组织，是否有本地临时实体或本地缓存。
- 消息入口：接收哪些 M2C，发送哪些 C2M，发送时机和失败处理。
- 表现事件：逻辑层如何发布事件给 HotfixView/Unity 表现层。
- UI/Prefab 绑定：使用哪些 Widget、Prefab、UIBindComponent、`UIButton`，不得在业务代码中直接 new UI 层级。
- 本地预测与校正：哪些行为先本地执行，哪些必须等服务端确认，服务端回包如何覆盖本地状态。
- 资源依赖：图集、Raw 资源、动画、音效、特效等只写逻辑依赖，具体美术需求放 `spec/art/`。

### 协议与配置必须双端一致

涉及协议或配置时必须说明：

- Proto 消息名、方向、字段、opcode 分配策略。
- 服务端生成文件与客户端生成文件都需要同步更新。
- Luban 配置表字段、服务端/客户端读取位置、默认值和兼容策略。
- 如果只改配置不改代码，需要明确生成脚本和二进制数据同步范围。

### 不应混入美术需求正文

功能逻辑文档可以引用美术需求，但不要把资源清单、拆图规格、尺寸格式、风格描述写成主体内容。此类内容应放入 `spec/art/`。
